= 4.6: Production security hardening

ChocoTech's Chief Information Security Officer (CISO) schedules an urgent meeting before the MaaS platform goes to production. "We need to ensure this AI infrastructure meets our security and compliance requirements," she explains. "We're subject to food safety regulations, and AI systems that impact quality control must be auditable and secure."

The security team raises several questions:

* "How do we verify model artifacts haven't been tampered with?"
* "Can we audit every API request to track who accessed which models?"
* "Are the models and data encrypted in transit and at rest?"
* "Does this meet SOC 2 and ISO 27001 requirements for AI systems?"

Before promoting the MaaS platform to production, you need to implement comprehensive security hardening across nine critical areas.

== Security hardening checklist

This checklist covers enterprise security requirements for production AI model serving platforms. Each section includes ChocoTech-specific implementation guidance.

=== 1. Model security

Ensure model artifacts are signed, verified, and tracked from source to deployment.

**Requirements:**

* ✓ Sign and verify model artifacts using sigstore or cosign
* ✓ Store models in a trusted registry or model repository
* ✓ Scan models for malicious code or unexpected behavior
* ✓ Use model provenance tracking (who trained, where, when, and how)
* ✓ Implement AIBOM (AI Bill of Materials) for transparency
* ✓ Monitor for model drift and adversarial inputs

**ChocoTech implementation:**

[source,role="execute"]
----
# Sign model artifact with cosign before uploading to S3
cosign sign-blob --key cosign.key llama3-8b-defect-analysis.safetensors > llama3-8b-defect-analysis.sig

# Verify signature before deployment
cosign verify-blob --key cosign.pub --signature llama3-8b-defect-analysis.sig llama3-8b-defect-analysis.safetensors
----

**Model provenance metadata:**

=== 2. Data security

Protect training data and inference requests with encryption and access controls.

**Requirements:**

* ✓ Encrypt data in transit (TLS/SSL)
* ✓ Encrypt data at rest (OpenShift + KMS integration)
* ✓ Ensure differential privacy or anonymization if using real user data
* ✓ Apply access controls on training and inference data

**ChocoTech implementation:**

**Encrypt data in transit:**

Verify all MaaS endpoints use TLS:

[source,role="execute"]
----
# Check MaaS gateway route uses edge TLS termination
oc get route maas-default-gateway -n redhat-ods-applications -o jsonpath='{.spec.tls.termination}'
----

Expected output:

----
edge
----

**Encrypt data at rest:**

Enable encryption for PersistentVolumes storing model weights:

[source,yaml]
----
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: model-storage-encrypted
  namespace: chocotech-models
  annotations:
    volume.kubernetes.io/storage-class: "encrypted-gp3"
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
  storageClassName: encrypted-gp3
----

Create an encrypted storage class with AWS KMS integration (example for AWS):

[source,yaml]
----
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: encrypted-gp3
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
  kmsKeyId: "arn:aws:kms:us-east-1:123456789:key/your-kms-key-id"
----

**Anonymize defect image metadata:**

Quality Control's defect images may contain PII in EXIF data (employee names, timestamps, location). Strip metadata before training:

[source,python]
----
from PIL import Image
import piexif

def anonymize_image(image_path):
    img = Image.open(image_path)
    # Remove all EXIF metadata
    img.save(image_path, exif=b"")
----

**Verification:**

* MaaS gateway routes use TLS edge termination
* PVCs for model storage use encrypted storage class
* Training data is anonymized or uses differential privacy
* API requests/responses are encrypted (no plaintext transmission)

**Input validation for prompt injection:**

Add input validation to prevent malicious prompts:

[source,python]
----
import re

def validate_prompt(user_input):
    # Block prompts attempting jailbreak or injection
    blocked_patterns = [
        r"ignore previous instructions",
        r"system: you are now",
        r"<\|im_start\|>system",
        r"</s><s>",  # Model control tokens
    ]

    for pattern in blocked_patterns:
        if re.search(pattern, user_input, re.IGNORECASE):
            raise ValueError("Input contains potentially malicious content")

    return True
----

=== 4. Runtime and infrastructure security

Isolate model serving workloads and monitor for anomalous behavior.

**Requirements:**

* ✓ Run models in isolated environments (containers with SELinux/seccomp/AppArmor)
* ✓ Use confidential containers or confidential computing (Intel SGX, AMD SEV) for sensitive models
* ✓ Monitor containers with runtime security tools (ACS - Advanced Cluster Security)
* ✓ Regularly patch and update base images and OS

**ChocoTech implementation:**

**Enable SELinux enforcement on model pods:**

Verify OpenShift nodes have SELinux enforcing:

[source,role="execute"]
----
oc debug node/<node-name> -- chroot /host getenforce
----

Expected output:

----
Enforcing
----

**Seccomp profile for model pods:**

Apply a restrictive seccomp profile to limit syscalls:

[source,yaml]
----
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama3-defect-secured
  namespace: chocotech-models
spec:
  # ... model config ...
  podSpec:
    securityContext:
      seccompProfile:
        type: RuntimeDefault
    containers:
      - name: vllm-container
        securityContext:
          allowPrivilegeEscalation: false
          runAsNonRoot: true
          capabilities:
            drop:
              - ALL
----

**Deploy Advanced Cluster Security (ACS) for runtime monitoring:**

NOTE: This requires Red Hat Advanced Cluster Security operator installed.

Create a policy to alert on unexpected model pod behavior:

[source,yaml]
----
apiVersion: policy.stackrox.io/v1
kind: SecurityPolicy
metadata:
  name: chocotech-model-runtime-policy
spec:
  description: "Alert on unexpected behavior in MaaS model pods"
  severity: HIGH
  policySection:
    - policyGroup: Anomalous Activity
  criteria:
    - namespace: chocotech-models
    - container:
        image: ".*vllm.*"
      runtime:
        processExecution:
          - "/bin/bash"  # Alert if shell is spawned in vLLM container
          - "/bin/sh"
          - "curl"       # Alert if outbound network tools used
----

Scan base images for CVEs before deployment:

* Model pods run with SELinux enforcing and seccomp profiles
* No privileged containers in chocotech-models namespace

=== 5. Authorization and access control

Implement role-based access control (RBAC) and multi-tenant isolation.

**Requirements:**

* ✓ Role-Based Access Control (RBAC) in OpenShift for model owners and users
* ✓ Implement multi-tenant isolation policies
* ✓ Use scoped tokens for API access with expiration

**ChocoTech implementation:**

**RBAC for model deployment:**

Create custom roles for model management:

[source,yaml]
----
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: model-deployer
  namespace: chocotech-models
rules:
  - apiGroups: ["serving.kserve.io"]
    resources: ["llminferenceservices"]
    verbs: ["create", "update", "patch", "delete", "get", "list"]
  - apiGroups: [""]
    resources: ["secrets", "configmaps"]
    verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: qc-team-model-deployers
  namespace: chocotech-models
subjects:
  - kind: Group
    name: qc-data-scientists
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: model-deployer
  apiGroup: rbac.authorization.k8s.io
----

**Multi-tenant namespace isolation:**

Enforce network policies to isolate departments:

[source,yaml]
----
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-cross-tenant-access
  namespace: chocotech-models
spec:
  podSelector: {}
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              name: chocotech-models
        - namespaceSelector:
            matchLabels:
              name: redhat-ods-applications
----

This policy ensures model pods only accept traffic from the MaaS gateway namespace, preventing direct cross-tenant access.

**API token scoping and expiration:**

Configure Kuadrant to enforce token expiration:

[source,yaml]
----
apiVersion: kuadrant.io/v1beta1
kind: AuthPolicy
metadata:
  name: token-expiration-policy
  namespace: redhat-ods-applications
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: maas-gateway-route
  rules:
    - jwt:
        issuerUrl: https://keycloak.chocotech.com/realms/maas
        ttl: 3600  # Tokens expire after 1 hour
----

**Verification:**

* Only authorized groups can deploy models
* Network policies isolate tenant namespaces
* API tokens have expiration (max 1 hour)
* RBAC prevents privilege escalation

**Requirements:**

* ✓ Enable audit logs for API access, model invocation, and data access
* ✓ Integrate with SIEM/logging (OpenShift + EFK stack, Splunk, Graylog)

**ChocoTech implementation:**

**Enable OpenShift API audit logging:**

[source,yaml]
----

  outputs:
    - name: splunk-maas-audit
      type: splunk
      url: https://splunk.chocotech.com:8088
      secret:
        name: splunk-hec-token
  pipelines:
    - name: forward-maas-audit
      inputRefs:
        - audit
        - application
      outputRefs:
        - splunk-maas-audit
      labels:
        namespace: "chocotech-models,redhat-ods-applications"
----

**Monitor for anomalous API usage:**

Create Prometheus alert for suspicious spike in model requests:

[source,yaml]
----
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: maas-anomaly-detection
  namespace: redhat-ods-applications
spec:
  groups:
    - name: maas-security-alerts
      interval: 1m
      rules:
        - alert: UnusualModelRequestSpike
          expr: |
            rate(http_requests_total{namespace="chocotech-models"}[5m]) >
            (avg_over_time(http_requests_total{namespace="chocotech-models"}[1h]) * 3)
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Unusual spike in model API requests detected"
            description: "Request rate is 3x higher than 1-hour average. Possible abuse or attack."
----

**Verification:**

* OpenShift audit logs capture all model deployment changes
* MaaS API requests are logged with user identity and timestamp
* Logs are forwarded to SIEM for long-term retention
* Anomaly detection alerts trigger for suspicious patterns

# MaaS Security Incident Response Playbook

## Incident Types

### 1. Model Compromise (Malicious model artifact)
- **Detection:** Signature verification failure, unexpected model behavior
- **Response:**
  1. Immediately scale LLMInferenceService to 0 replicas
  2. Quarantine compromised model artifact
  3. Alert security team (security@chocotech.com)
  4. Forensic analysis: check S3 access logs, container image provenance

### 2. Data Exfiltration (Unauthorized API access)

- **Response:**

  2. Review Kuadrant Authorino audit logs

- **Response:**

  2. Identify attacking user/tier via logs
  3. Suspend attacker's API access
  4. Scale gateway replicas if infrastructure-level DoS

**Issue:** Cosign signature verification fails with "signature not found" error

**Solution:**

. Verify signature file exists: `ls -la llama3-8b-defect-analysis.sig`
. Check public key matches private key used for signing: `cosign public-key --key cosign.key`
. Ensure signature was generated for the exact file (checksum match): `sha256sum llama3-8b-defect-analysis.safetensors`
. Re-sign if signature is stale or keys rotated

**Issue:** OpenShift audit logging generates excessive log volume (10GB+/day)

**Solution:**

. Reduce audit profile from `WriteRequestBodies` to `Default`: `oc patch apiserver cluster --type=merge -p '{"spec":{"audit":{"profile":"Default"}}}'`
. Filter audit logs to MaaS-related namespaces only in ClusterLogForwarder
. Use Prometheus recording rules to aggregate metrics instead of retaining raw logs
. Implement log retention policy (30 days for audit, 7 days for application logs)

**Issue:** ACS reports policy violation for "privileged container" but no privileged containers are defined

**Solution:**

. Check pod security context: `oc get pod <pod-name> -n chocotech-models -o yaml | grep -A 5 securityContext`
. Common false positive: GPU device plugin requires privileged init container (expected)
. Whitelist GPU operator namespace in ACS policy: `nvidia-gpu-operator`
. Verify issue is with model pod, not infrastructure pods

**Issue:** Sealed secret cannot be decrypted by cluster

**Solution:**

. Verify sealed-secrets controller is running: `oc get pods -n kube-system | grep sealed-secrets`
. Check sealed secret was encrypted for correct cluster: `kubeseal --fetch-cert` (compare cert)
. Ensure sealed secret is in correct namespace (sealed secrets are namespace-scoped)
. Re-encrypt secret if sealed-secrets controller was reinstalled (new encryption key)
