== Exercise 1: Understanding the GPU silo crisis

ChocoTech Industries has four teams independently managing GPU infrastructure. Let's examine the operational pain points this creates.

You begin by reviewing the current state of GPU allocations across ChocoTech's business units.

=== The problem with dedicated accelerators

**Current state at ChocoTech**:

* **Quality Control**: 2x NVIDIA A100 GPUs running defect detection models
* **R&D**: 4x NVIDIA L40S GPUs for recipe optimization
* **Marketing**: 2x NVIDIA L4 GPUs for sentiment analysis
* **Supply Chain**: 2x NVIDIA L40S GPUs for inventory prediction

**Operational pain points identified**:

**GPU under-utilization and over-provisioning**:

Teams request and manage their own dedicated GPUs, leading to expensive hardware sitting under-utilized while other high-priority projects may be stalled or additional hardware is provisioned unnecessarily.

*ChocoTech example*: Quality Control's A100 GPUs process defect images only during production shifts (8 hours/day), leaving $120,000 of hardware idle 67% of the time. Meanwhile, R&D waits three weeks for GPU capacity to test new flavor prediction models.

**Misallocation of engineer and data science expertise**:

Data scientists spend more time debugging CUDA drivers, Docker containers, and Kubernetes manifests than actually tuning models or analyzing data.

*ChocoTech example*: Marketing's data science team spent 40 hours last quarter troubleshooting vLLM container startup issues instead of improving their customer sentiment models.

**Redundant model deployments**:

Five different teams download and deploy the same generative AI model, leading to overall waste and duplication when they could be sharing resources efficiently.

*ChocoTech example*: All four teams independently deployed Llama-3 8B for different use cases, consuming 128GB of storage (4 x 32GB) and wasting GPU memory with duplicate model weights.

**Security "black boxes"**:

Without a centralized API gateway, there is zero visibility into who is sending what data to which model, making audit logs and PII filtering nearly impossible to enforce.

*ChocoTech example*: Marketing's sentiment analysis application processes customer feedback containing personally identifiable information (PII), but there's no way to audit which users accessed which models or implement PII filtering policies.

**No way to track usage or forecast needs**:

Without the ability to track model usage across users, applications, and different models, there is no way to forecast compute needs accurately.

*ChocoTech example*: Finance cannot justify GPU infrastructure expansion requests because there's no data showing which teams are consuming capacity, which models are heavily used, or whether existing hardware is properly utilized.

=== The solution - Centralized model hub with MaaS

Models as a Service (MaaS) transforms GPU infrastructure from dedicated, single-team clusters into a consumable model backplane.

**How MaaS changes the operating model**:

**Central IT (Platform Engineers)**::
Manages the inference strategy, hardware allocation, and governance policies. Controls which models are available, enforces access tiers, and monitors infrastructure health.

**Lines of Business (Marketing, R&D, Quality Control, Supply Chain)**::
Consume AI capabilities via tokens, treating model inference as a metered, on-demand internal service. Focus on building applications, not managing infrastructure.

**Platform Engineers**::
Configure tiers, rate limits, quotas, and observability dashboards. Ensure fair resource allocation and compliance with organizational policies.

**Developers**::
Integrate models via OpenAI-compatible APIs without worrying about GPU drivers, Kubernetes manifests, or model serving runtimes.

**Business impact realized**:

**Without MaaS**:

* Redundant model deployments waste storage and compute
* No audit trail for compliance requirements
* Cannot track usage or forecast future needs
* Teams compete for scarce GPU resources

**With MaaS**:

* Centralized model catalog eliminates redundancy
* Complete audit logs and PII filtering via API gateway
* Token-based consumption enables accurate chargeback
* Fair resource allocation prevents team conflicts

== Exercise 2: The MaaS architecture stack

Now that you understand the business problem, let's explore the technical architecture that makes MaaS possible.

MaaS on Red Hat OpenShift AI is built on three distinct layers, each with specific responsibilities. Understanding this layered architecture is critical for effective configuration and troubleshooting.

=== The three-layer architecture

A user request for model inference passes through three distinct layers:

----
User Request → Gateway API → Kuadrant → KServe → Model
             (Routing)     (Policy)   (Serving)
----

Each layer has a specific responsibility and solves a distinct problem:

**Gateway API (Layer 1 - Traffic routing)**::
Network traffic routing and ingress control. Exposes a single API endpoint that routes to multiple backend models. Handles TLS termination and path-based routing.

**Kuadrant (Layer 2 - Policy enforcement)**::
Authorization (Authorino) and rate limiting (Limitador). Enforces tier-based access control, token quotas, and usage tracking. Returns 401/403/429 status codes for policy violations.

**KServe (Layer 3 - Model serving)**::
Model serving infrastructure. Manages model lifecycle, distributed inference with LeaderWorkerSet, autoscaling, and health checks.

**Why this layered architecture matters**:

**Separation of concerns**::
Each component has one job. KServe doesn't do authentication. Kuadrant doesn't do model serving. Gateway API doesn't enforce quotas.

**Flexibility**::
Replace Gateway API implementation without changing KServe. Adjust rate limits without redeploying models. Swap serving runtimes (vLLM, TGI) without touching policy layer.

**Auditability**::
Every request passes through Kuadrant, creating a complete audit trail with user identity, model accessed, tokens consumed, and timestamp.

**Chargeback accuracy**::
Kuadrant's token tracking enables precise cross-department billing based on actual consumption rather than infrastructure allocation.

=== Layer 3 - KServe (model serving)

KServe provides the foundation for serving generative AI models on OpenShift.

**Core capabilities**:

**Model deployment**::
Packages models in containers with serving runtimes (vLLM, llm-d, Hugging Face TGI). Abstracts complexity of GPU allocation and CUDA drivers.

**Distributed inference with LeaderWorkerSet**::
Splits large models across multiple GPUs using tensor parallelism. Critical for models exceeding single GPU memory capacity (70B+ parameter models).

**Autoscaling**::
Scales model replicas based on request load. Scales to zero when idle to conserve GPU resources. Scales up when inference demand increases.

**Health checks and readiness**::
Monitors model container health with liveness and readiness probes. Ensures requests only route to healthy model instances.

**What KServe does NOT do**:

* Authentication or authorization (handled by Kuadrant/Authorino)
* Rate limiting or quota enforcement (handled by Kuadrant/Limitador)
* API routing or ingress (handled by Gateway API)

A Platform Engineer deploys a model using KServe. KServe ensures it's running, healthy, and can handle inference requests. But KServe alone doesn't control who can access the model or how often.

=== Layer 2 - Kuadrant (policy enforcement)

Kuadrant adds governance and policy enforcement on top of KServe model serving.

**Authorino - Authentication and authorization**:

* Validates Bearer tokens from API requests
* Maps OpenShift users to tier levels (bronze, silver, gold, platinum)
* Returns `401 Unauthorized` for invalid or missing tokens
* Returns `403 Forbidden` when user's tier cannot access a specific model
* Integrates with OpenShift OAuth for user identity

**Limitador - Rate limiting and quotas**:

* Enforces requests-per-minute limits based on user tier
* Tracks token consumption quotas (e.g., 10,000 tokens/hour for bronze tier)
* Returns `429 Rate Limit Exceeded` when limits are hit
* Includes `Retry-After` header for client backoff logic
* Provides metrics for chargeback and capacity planning

**Why Kuadrant is critical for enterprise AI**:

Without Kuadrant, any user with cluster access could consume unlimited GPU resources, monopolize capacity, and prevent other teams from accessing models. Kuadrant enforces fair resource allocation and provides the audit trail required for compliance.

*ChocoTech example*: Marketing team is assigned "bronze" tier with 1,000 requests/hour. When they exceed this limit during a campaign launch, Kuadrant returns `429` status, preventing them from starving R&D team's "gold" tier access.

=== Layer 1 - Gateway API (traffic routing)

Gateway API exposes MaaS to external clients and routes traffic to the policy and serving layers.

**Core capabilities**:

**Single endpoint for all models**::
One URL (`https://maas-default-gateway.{cluster_domain}`) for all models. Eliminates need for per-model hostnames or complex DNS management.

**Path-based routing**::
`/v1/models` lists available models, `/v1/chat/completions` routes to specific models based on request payload. OpenAI-compatible API paths for developer familiarity.

**TLS termination**::
Handles HTTPS certificates (via cert-manager or OpenShift Routes). Ensures encrypted communication between clients and API gateway.

**Request forwarding**::
Routes validated requests from Gateway → Kuadrant (policy check) → KServe (model inference) → response back to client.

**Integration with Istio Service Mesh**::
Gateway API uses Istio as the underlying implementation, providing traffic management, observability, and resiliency features.

=== The complete request flow

Here's what happens when a developer at ChocoTech queries a model from their sentiment analysis application:

----
1. Developer sends: POST /v1/chat/completions
   Headers: Authorization: Bearer <token>
   Body: {"model": "llama-3-8b", "messages": [...]}

2. Gateway API receives request at maas-default-gateway endpoint

3. Kuadrant/Authorino validates Bearer token
   - Invalid token? Return 401 Unauthorized (stop here)
   - Valid token? Extract user identity and tier level

4. Kuadrant/Authorino checks model access policy
   - User's tier cannot access llama-3-8b? Return 403 Forbidden (stop here)
   - User's tier has access? Proceed to rate limit check

5. Kuadrant/Limitador checks rate limits and quotas
   - Exceeded requests/minute? Return 429 Rate Limit Exceeded (stop here)
   - Exceeded token quota? Return 429 with quota message (stop here)
   - Within limits? Forward request to KServe

6. Request forwarded to KServe InferenceService

7. KServe routes to appropriate model pod (llama-3-8b-predictor-xxx)

8. vLLM serving runtime generates inference response

9. Response flows back: KServe → Gateway API → Developer's application

10. Kuadrant logs request metadata for chargeback:
    - User: marketing-dev-user
    - Model: llama-3-8b
    - Tokens consumed: 247 (input: 50, output: 197)
    - Timestamp: 2026-04-28T14:32:17Z
    - Status: 200 OK
----

This layered approach ensures that every request is authenticated, authorized, rate-limited, audited, and routed to the correct model serving infrastructure.

=== Verify the architecture layers

Now let's verify that the three architectural layers are installed and healthy in your OpenShift cluster.

Check KServe components (model serving layer):

[source,role="execute"]
----
oc get pods -n redhat-ods-applications | grep -E "(kserve|llm)"
----

Expected output (pods should be Running):

----
kserve-controller-manager-xxx           1/1     Running
llm-serving-runtime-controller-xxx      1/1     Running
----

Check Kuadrant components (policy enforcement layer):

[source,role="execute"]
----
oc get pods -n kuadrant-system
----

Expected output:

----
NAME                          READY   STATUS    RESTARTS   AGE
authorino-xxx                 1/1     Running   0          5d
limitador-xxx                 1/1     Running   0          5d
----

Check Gateway API resources:

[source,role="execute"]
----
oc get gateway -n redhat-ods-applications
----

Expected output:

----
NAME                   CLASS   ADDRESS                              PROGRAMMED   AGE
maas-default-gateway   istio   maas-default-gateway.example.com    True         5d
----

Verify Service Mesh (Istio) integration:

[source,role="execute"]
----
oc get pods -n istio-system
----

Expected output (Istio control plane):

----
istiod-xxx                    1/1     Running
istio-ingressgateway-xxx      1/1     Running
----

=== Verification

Confirm that all three layers are healthy:

* KServe controller pods are Running in redhat-ods-applications namespace
* Kuadrant pods (authorino, limitador) are Running in kuadrant-system namespace
* Gateway resource exists with PROGRAMMED=True status
* Istio control plane pods are Running in istio-system namespace

If any components are missing or unhealthy, the MaaS platform cannot function correctly. Each layer depends on the others for complete end-to-end model serving with governance.

== Exercise 3: Hardware and platform prerequisites

MaaS builds on top of generic GPU enablement. If GPU drivers, operators, and hardware certification aren't in place, model serving will fail.

== Troubleshooting

**Issue**: KServe controller pods not found in redhat-ods-applications namespace

**Solution**:

. Check DataScienceCluster KServe management state: `oc get datasciencecluster default-dsc -o yaml | grep -A 3 kserve`
. If `managementState: Removed`, KServe is disabled
. Module 2 will cover enabling KServe through the DataScienceCluster

**Issue**: Kuadrant pods not found in kuadrant-system namespace

**Solution**:

. Kuadrant is deployed when MaaS is enabled in DataScienceCluster
. This is expected if MaaS hasn't been configured yet
. Module 2 covers enabling the MaaS component

**Issue**: GPU nodes not showing in `oc get nodes` output

**Solution**:

. Verify GPU operator installation: `oc get pods -n nvidia-gpu-operator`
. Check node labels manually: `oc get nodes --show-labels | grep nvidia`
. Ensure GPU hardware is properly installed and detected by the host OS

**Issue**: Gateway resource returns "PROGRAMMED=False"

**Solution**:

. Check Gateway status for error messages: `oc describe gateway -n redhat-ods-applications`
. Verify Istio Service Mesh control plane is healthy: `oc get pods -n istio-system`
. Common cause: Service Mesh operator not installed or control plane not deployed

**Issue**: Service Mesh control plane shows "ComponentsNotReady"

**Solution**:

. Check Service Mesh operator logs: `oc logs -n openshift-operators deployment/istio-operator`
. Verify ServiceMeshControlPlane resource: `oc get smcp -n istio-system -o yaml`
. Ensure required network policies and security constraints are configured
