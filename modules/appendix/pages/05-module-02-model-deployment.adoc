= 3.1: Understanding serving runtimes and distributed inference

Before deploying a model, you need to understand the difference between standalone KServe deployment and MaaS-integrated deployment.

Only models deployed using the **Distributed Inference with llm-d** runtime can integrate with MaaS governance and appear in the Gen AI Studio catalog.

=== Standalone InferenceService vs MaaS-published model

**Standalone KServe InferenceService**:

* Bypasses MaaS gateway entirely
* No tier-based access control or authentication
* No token consumption tracking for chargeback
* No rate limiting or quotas enforcement
* Direct model access without Bearer token
* Cannot appear in Gen AI Studio catalog
* Suitable for internal testing or non-governed workloads

**MaaS-published model (Distributed Inference with llm-d)**:

* Routes through Gateway API and Kuadrant policy layer
* Enforces tier-based access control (bronze/silver/gold/platinum)
* Tracks token consumption for accurate chargeback
* Applies request rate limits and quotas per user tier
* Requires Bearer token authentication for all API calls
* Appears in Gen AI Studio catalog automatically
* Suitable for production, multi-tenant, governed model serving

IMPORTANT: If you deploy a model using any runtime other than "Distributed Inference with llm-d" (such as standard KServe, vLLM without llm-d, NVIDIA NIM, or MLServer), the option to "Publish as MaaS endpoint" will not appear in the dashboard.

=== Why distributed inference matters for ChocoTech

ChocoTech needs to deploy both small models (Llama-3 8B) and large models (Nemotron 70B) efficiently.

**Small models (8B parameters)**:

* Fit on a single GPU (24GB VRAM sufficient)
* Can use single-replica deployment
* Distributed inference still recommended for MaaS integration

**Large models (70B+ parameters)**:

* Exceed single GPU memory capacity
* Require tensor parallelism across multiple GPUs
* Must use distributed inference with LeaderWorkerSet

**LeaderWorkerSet architecture**:

----
Leader Pod (Coordinator)       Worker Pods (Inference)
┌─────────────────┐           ┌───────────────┐
│ Receives requests│───────────│ GPU 1 (shard 1)│
│ Coordinates work │           ├───────────────┤
│ Aggregates output│───────────│ GPU 2 (shard 2)│
└─────────────────┘           ├───────────────┤
                              │ GPU 3 (shard 3)│
                              └───────────────┘
----

The leader pod coordinates inference requests while worker pods process model shards in parallel across multiple GPUs.

=== Verify distributed inference components

Check that LeaderWorkerSet operator is installed:

[source,role="execute"]
----
oc get pods -n leaderworkerset-system
----

Expected output:

----
NAME                                            READY   STATUS    RESTARTS   AGE
leaderworkerset-controller-manager-xxx          1/1     Running   0          5d
----

If LeaderWorkerSet operator is not installed, distributed inference cannot function.

Check llm-d serving runtime controller:

[source,role="execute"]
----
oc get pods -n redhat-ods-applications | grep llm-serving
----

Expected output:

----
llm-serving-runtime-controller-xxx   1/1     Running   0          5d
----

=== Verification

Confirm distributed inference prerequisites:

* LeaderWorkerSet operator is Running in leaderworkerset-system namespace
* llm-serving-runtime controller is Running in redhat-ods-applications namespace
* GPU nodes are available with sufficient VRAM for model deployment
== Exercise 3: Deploy your first model with distributed inference

Now you'll deploy Llama-3 8B using the LLMInferenceService custom resource for ChocoTech's R&D team.

The LLMInferenceService CRD is the declarative way to deploy models with distributed inference and MaaS integration.

=== Create a project for model deployment

Create a dedicated namespace for model serving:

[source,role="execute"]
----
oc new-project chocotech-models
----

Label the project to enable MaaS integration:

[source,role="execute"]
----
oc label namespace chocotech-models modelsAsService=true
----

This label tells the MaaS operator to watch for LLMInferenceService resources in this namespace.

=== Prepare the LLMInferenceService manifest

Create a YAML file for Llama-3 8B deployment:

[source,role="execute"]
----
cat > llama3-8b-llminferenceservice.yaml <<EOF
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama-3-8b
  namespace: chocotech-models
  labels:
    app: llama-3-8b
spec:
  model:
    name: meta-llama/Meta-Llama-3-8B-Instruct
    source: huggingface
  runtime:
    name: vllm
    version: v0.5.4
  resources:
    limits:
      nvidia.com/gpu: 1
    requests:
      nvidia.com/gpu: 1
      memory: 16Gi
  scaling:
    minReplicas: 1
    maxReplicas: 3
  maasConfig:
    publishToGenAIStudio: true
    tierAccess:
      - bronze
      - silver
      - gold
      - platinum
EOF
----

**Key fields explained**:

* `spec.model.name` - Hugging Face model identifier
* `spec.model.source` - Model source (huggingface, S3, PVC)
* `spec.runtime.name` - Serving runtime (vllm with llm-d integration)
* `spec.resources.limits` - GPU and memory allocation
* `spec.scaling` - Autoscaling configuration (min/max replicas)
* `spec.maasConfig.publishToGenAIStudio` - Publish to catalog (true/false)
* `spec.maasConfig.tierAccess` - Which user tiers can access this model

=== Deploy the model

Apply the LLMInferenceService manifest:

[source,role="execute"]
----
oc apply -f llama3-8b-llminferenceservice.yaml
----

Expected output:

----
llminferenceservice.serving.kserve.io/llama-3-8b created
----

=== Monitor model deployment

Watch the LLMInferenceService status:

[source,role="execute"]
----
oc get llminferenceservice -n chocotech-models -w
----

You'll see the status progress through phases:

----
NAME          READY   REASON              AGE
llama-3-8b    False   Initializing        10s
llama-3-8b    False   DownloadingModel    45s
llama-3-8b    False   LoadingModel        2m
llama-3-8b    True    Ready               5m
----

Press `Ctrl+C` once status shows `READY=True`.

Watch the underlying pods being created:

[source,role="execute"]
----
oc get pods -n chocotech-models -w
----

You'll see LeaderWorkerSet creating leader and worker pods:

----
NAME                                    READY   STATUS              RESTARTS   AGE
llama-3-8b-0-0                          0/1     ContainerCreating   0          30s
llama-3-8b-0-0                          0/1     PodInitializing     0          1m
llama-3-8b-0-0                          1/1     Running             0          3m
----

Press `Ctrl+C` once pods are Running.

=== Verify model deployment

Check that the model is healthy:

[source,role="execute"]
----
oc get llminferenceservice llama-3-8b -n chocotech-models -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}'
----

Expected output:

----
True
----

Check the model service was created:

[source,role="execute"]
----
oc get svc -n chocotech-models | grep llama
----

Expected output:

----
llama-3-8b-predictor-00001   ClusterIP   172.30.xxx.xxx   <none>        8080/TCP   5m
----

Check model logs for successful startup:

[source,role="execute"]
----
oc logs -n chocotech-models -l app=llama-3-8b --tail=20
----

Look for log messages indicating successful model loading:

----
INFO:     vLLM v0.5.4
INFO:     Initializing engine for meta-llama/Meta-Llama-3-8B-Instruct
INFO:     Model loaded successfully on GPU 0
INFO:     Uvicorn running on 0.0.0.0:8080
----

=== Verification

Confirm model deployment:

* LLMInferenceService status shows READY=True
* Leader pod is Running with 1/1 ready containers
* Model service exists in chocotech-models namespace
* Model logs show successful initialization and GPU loading
* vLLM serving runtime is active on port 8080
