# RHOAI Pattern Validation Checklist

Use this checklist BEFORE finalizing any generated content. This ensures the output matches the 13 RHOAI stylistic signatures and will pass verify-rhoai-content validation.

## Pre-Publication Validation

Run through this checklist before outputting the final AsciiDoc. If any item is not met, revise.

### 1. Lead Opener (CRITICAL)

- [ ] Present: Section starts with `[.lead]` block
- [ ] Format: Content is `*bold*` (asterisks)
- [ ] Content: Italic formatting applied (bold + italic: `[.lead]\n*...*`)
- [ ] Tone: Single sentence, declarative statement
- [ ] Authority: Establishes engineering credibility
- [ ] NOT vague: Not "this section covers..." or "we will learn..."
- [ ] NOT a question: Not phrased as question
- [ ] Unique: Different from section title

**Example Test**:
- ✅ `[.lead]\n*Inference is where the bill arrives.*`
- ❌ `[.lead]\nThis section covers inference costs.`
- ❌ `[.lead]\n*Have you thought about inference?*`

### 2. Business Value Connection

- [ ] Present: Business context paragraph follows lead opener
- [ ] Timing: Within first 100 words of content
- [ ] Specificity: Mentions concrete business concern (cost, compliance, efficiency, reliability)
- [ ] Numbers: Includes specific numbers/percentages where relevant
- [ ] Pattern: Uses "developer vs platform engineer" or similar contrast
- [ ] Relevance: Directly connects to learning objective
- [ ] Scenario: If ACME scenario provided, integrated naturally (not forced)
- [ ] Platform focus: Speaks from platform engineer perspective (not developer)

**Example Test**:
- ✅ "A developer downloads models freely. A platform engineer sees GPU costs and compliance risks."
- ❌ "Machine learning is important in modern organizations."
- ❌ Generic description without business context

### 3. Document Attributes

- [ ] time_estimate present: `:time_estimate: XX`
- [ ] Title present: `= Section Title`
- [ ] Reading time shown: `_Estimated reading time: *{time_estimate} minutes*._`
- [ ] Time realistic: 15-20 min (presentation), 30-60 min (lab), 10-15 min (quiz)

### 4. Objective Definition

- [ ] Syntax correct: `Objective::` (with ::)
- [ ] Content specific: Describes learner outcome, not the section
- [ ] Aligned with learning_objective parameter: Matches what was requested
- [ ] Measurable: Can be validated by learner

**Example Test**:
- ✅ `Objective::\nDeploy a fine-tuned model to production with governance controls.`
- ❌ `Objective::\nLearn about deployment.`

### 5. Definition Lists for Structure

- [ ] Objective:: used correctly
- [ ] Definition list syntax (::) used for key concepts
- [ ] Not using bullets when definition lists appropriate
- [ ] At least 3-5 key definitions in presentation/lab

### 6. Content Structure (Business Value → Architecture → Implementation)

For presentations/labs, check order:
1. [ ] Business Value section comes first (what, why, cost)
2. [ ] Architecture section explains how it works
3. [ ] Implementation section shows how to do it
4. [ ] Validation section shows how to verify

(Quizzes have different structure—see quiz-specific checks below.)

### 7. Code Block Formatting

For all code blocks:
- [ ] Source type specified: `[source,bash]`, `[source,yaml]`, etc.
- [ ] Correct type used:
  - [ ] Commands: `[source,bash]` or `[source,console]`
  - [ ] YAML: `[source,yaml]`
  - [ ] Python: `[source,python]`
  - [ ] Output: No source type (plain text)
- [ ] Syntax valid: Will parse without errors
- [ ] Copy-paste ready: No placeholders without context

For lab sections specifically:
- [ ] Verification steps included after deployments
- [ ] Expected output shown (not just commands)
- [ ] Success criteria clear (✓ checkmarks indicate success)

### 8. What's Next Section (CRITICAL)

- [ ] Present: "== What's Next" section exists
- [ ] Not generic: Not "Next we'll learn about X"
- [ ] Business-focused: Explains what's enabled or improved
- [ ] Transitions flow: Connects logically to next topic (check if this is last section)
- [ ] Forward-looking: Motivates next section

**Example Test**:
- ✅ "Now that you understand the cost, the next section shows how to optimize VRAM usage to reduce your bill by 40%."
- ❌ "In the next section, you will learn about VRAM optimization."

### 9. Prohibited Terms

Search and REMOVE any of these:
- [ ] "robust"
- [ ] "powerful"
- [ ] "leverage" (use "use" instead)
- [ ] "synergy"
- [ ] "game-changer"
- [ ] "cutting-edge"
- [ ] "best-in-class"
- [ ] "innovative" (be specific instead)

### 10. Voice & Tone

- [ ] Declarative: Uses "is", "must", not "might", "could"
- [ ] Active voice: "You deploy" not "It is deployed"
- [ ] Direct: Addresses reader as "you"
- [ ] Specific: Uses numbers, percentages, concrete examples
- [ ] Engineering authority: Assumes reader is smart and busy
- [ ] No hedging: Confident assertions, not tentative language

### 11. Capitalization & Grammar

- [ ] Headlines: Sentence case, not Title Case
  - ✅ "Understanding GPU memory requirements"
  - ❌ "Understanding GPU Memory Requirements"
- [ ] Product names: "Red Hat OpenShift AI" (first use), then "OpenShift AI" or "RHOAI"
- [ ] No "The" before product: "OpenShift AI" not "the OpenShift AI"
- [ ] Oxford commas: "models, inference, and deployment" (serial comma)
- [ ] Numbers: <10 spelled out ("five nodes"), >=10 as numerals ("12 instances")

### 12. Lab-Specific Checks (if section_type == "lab")

- [ ] Prerequisites listed: "Before You Begin" section
- [ ] Setup instructions clear: How to start the lab
- [ ] Numbered tasks: 2-4 major exercises
- [ ] Each task has verification: Command showing success
- [ ] Expected output shown: Not just the command
- [ ] Verification checkmark: "✓ [What this proves]"
- [ ] Troubleshooting section: If include_troubleshooting=true
  - [ ] 2-3 common failure modes
  - [ ] Each has diagnosis and solution
  - [ ] Relevant to the lab (not generic)
- [ ] Learning outcomes checklist: Learner can self-assess
- [ ] Realistic time estimate: 45-60 min for full lab

### 13. Quiz-Specific Checks (if section_type == "quiz")

- [ ] 5 questions provided: No fewer, can be up to 6
- [ ] Scenario-based: Each has business context/scenario
- [ ] Multiple choice: 3-4 options per question
- [ ] Correct answer marked: [x] on right answer
- [ ] Explanation provided: Why this answer is correct
- [ ] Wrong answers explained: Why other options don't work
- [ ] Scoring guide: How to interpret results
- [ ] Review areas: If low score, what to study
- [ ] Questions test decisions: Not just memorization

### 14. Presentation-Specific Checks (if section_type == "presentation")

- [ ] Multiple sections: At least 2-3 major topics (== headings)
- [ ] Trade-offs discussed: Explains why one approach over another
- [ ] Key Concepts Summary: Bullet list of 4-6 concepts with business impact
- [ ] Common Mistakes section: 1-2 mistakes to avoid + how to fix
- [ ] Realistic time estimate: 15-20 min to read

## Quality Gate: Will This Pass verify-rhoai-content?

This skill aims to generate content scoring 85+ (GOOD or STRONG).

**Scoring Breakdown** (what verify-rhoai-content checks):

- **Structure** (25%):
  - ✅ Title, lead, objective, sections, what's next all present
  - ✅ Proper hierarchy (no skipped heading levels)
  - ✅ Proper document attributes

- **RHOAI Style** (35%):
  - ✅ Lead opener present and strong (25% of this subscore)
  - ✅ Business narrative strong (30%)
  - ✅ Definition lists used (20%)
  - ✅ What's Next section strong (15%)
  - ✅ Technical authority/specificity (10%)

- **Style Guide** (20%):
  - ✅ No prohibited terms
  - ✅ Sentence case headlines
  - ✅ Inclusive language
  - ✅ Proper capitalization and grammar

- **Technical** (15%):
  - ✅ Code blocks correct syntax
  - ✅ Product names official
  - ✅ Commands valid (if lab)
  - ✅ No misleading technical claims

- **Accessibility** (5%):
  - ✅ Alt text on images (if any)
  - ✅ Descriptive links
  - ✅ Proper heading hierarchy

**To hit 85+:**
- Must have: Strong lead opener, clear business value, definition lists, solid what's next
- Can afford: Minor formatting tweaks, a couple of capitalization issues
- Cannot afford: Missing sections, vague business connection, poor what's next

## Final Checklist Before Output

Before you finish, answer these:

1. **Is this a complete AsciiDoc section?** (Not a template, actual content)
   - [ ] Yes, ready to build with Antora

2. **Does it solve the learning objective?**
   - [ ] Yes, learner can accomplish the stated goal

3. **Does it follow RHOAI patterns exactly?**
   - [ ] Lead opener, business value, definition lists, what's next all present

4. **Is it ready for publication without edits?**
   - [ ] Yes, all content filled in, no placeholders

5. **Will it score 85+ on verify-rhoai-content?**
   - [ ] Confidence: HIGH / MEDIUM / LOW
   - [ ] If LOW, review and revise before outputting

---

## What to Do If Something's Missing

**If lead opener is missing/weak**: Revise to single declarative sentence matching examples

**If business value isn't connected**: Add paragraph connecting to cost/compliance/efficiency/reliability

**If What's Next is generic**: Rewrite to explain what's enabled, not just what's next

**If code blocks aren't verified**: Add expected output section after each command

**If time estimate seems wrong**: Adjust based on section length and complexity

**If prohibited terms appear**: Search/replace with specific language

**If checklist item fails**: Fix it before outputting—this is non-negotiable for quality
