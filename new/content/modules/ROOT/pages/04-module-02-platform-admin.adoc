== Exercise 1: Enable MaaS in DataScienceCluster

The `DataScienceCluster` custom resource controls which OpenShift AI components are active. You'll update this CRD to enable MaaS.

OpenShift AI uses a declarative configuration model. You declare desired state in CRDs, and the operator reconciles actual state to match. This differs from imperative deployment where you manually create pods, services, and configurations.

=== Understanding the DataScienceCluster CRD

The DataScienceCluster CRD has a `components` section listing all available features:

* `kserve` - Model serving infrastructure (prerequisite for MaaS)
* `modelsAsService` - MaaS API, governance, and Gateway integration
* `dashboard` - OpenShift AI web console
* Other components like Pipelines, Workbenches, Model Registry

Each component has a `managementState` field:

* `Managed` - Operator deploys and maintains the component
* `Removed` - Operator does not deploy the component

=== Check current DataScienceCluster state

View the current configuration:

[source,role="execute"]
----
oc get datasciencecluster default-dsc -o yaml
----

Look for the `components` section. You'll likely see:

----
spec:
  components:
    kserve:
      managementState: Managed
    modelsAsService:
      managementState: Removed
    dashboard:
      managementState: Managed
----

Note that `modelsAsService` is currently `Removed`, meaning MaaS is not deployed.

=== Enable MaaS component

Edit the DataScienceCluster:

[source,role="execute"]
----
oc edit datasciencecluster default-dsc
----

Find the `modelsAsService` section and change `managementState` from `Removed` to `Managed`:

----
spec:
  components:
    modelsAsService:
      managementState: Managed  # Change this from "Removed" to "Managed"
    kserve:
      managementState: Managed  # Ensure this is also Managed
    dashboard:
      managementState: Managed
----

Save and exit the editor (`:wq` in vi, or `Ctrl+X` then `Y` in nano).

=== Monitor MaaS deployment

Watch the operator deploy MaaS components:

[source,role="execute"]
----
oc get pods -n redhat-ods-applications -w | grep maas
----

You should see pods being created:

----
maas-api-xxx                  0/1     ContainerCreating   0          5s
maas-default-gateway-xxx      0/1     Init:0/1            0          5s
----

After 2-5 minutes, all pods should reach `Running` state:

----
maas-api-xxx                  1/1     Running   0          3m
maas-default-gateway-xxx      1/1     Running   0          3m
----

Press `Ctrl+C` to stop watching.

=== Verify MaaS API deployment

Check that the MaaS API pod is healthy:

[source,role="execute"]
----
oc get pods -n redhat-ods-applications | grep maas-api
----

Expected output:

----
maas-api-7c8f9d5b-xxx         1/1     Running   0          5m
----

Check the MaaS API service:

[source,role="execute"]
----
oc get svc -n redhat-ods-applications | grep maas
----

Expected output:

----
maas-api                ClusterIP   172.30.xxx.xxx   <none>        8080/TCP   5m
maas-default-gateway    ClusterIP   172.30.xxx.xxx   <none>        80/TCP     5m
----

=== Verification

Confirm MaaS enablement:

* DataScienceCluster shows `modelsAsService.managementState: Managed`
* MaaS API pod is Running with 1/1 ready containers
* MaaS service exists in redhat-ods-applications namespace
* Gateway pod is Running and ready

== Exercise 2: Enable Gen AI Studio dashboard

Now that the MaaS backend is deployed, you need to activate the dashboard interface so users can discover models and generate API tokens.

The `OdhDashboardConfig` CRD controls which features appear in the OpenShift AI web console.

=== Understanding the dashboard configuration

The OdhDashboardConfig enables or disables UI features:

* `genAiStudio` - Gen AI Studio menu item and model catalog UI
* `modelAsService` - MaaS-specific dashboard pages (tier management, usage metrics)
* `modelServing` - Traditional KServe model serving UI
* Other features like Data Science Projects, Workbenches, Pipelines

=== Check current dashboard configuration

View the current OdhDashboardConfig:

[source,role="execute"]
----
oc get odhdashboardconfig odh-dashboard-config -o yaml
----

Look for the `dashboardConfig` section:

----
spec:
  dashboardConfig:
    genAiStudio: false
    modelAsService: false
    disableModelServing: false
----

Note that both `genAiStudio` and `modelAsService` are currently false.

=== Enable Gen AI Studio features

Edit the OdhDashboardConfig:

[source,role="execute"]
----
oc edit odhdashboardconfig odh-dashboard-config
----

Update the `dashboardConfig` section:

----
spec:
  dashboardConfig:
    genAiStudio: true           # Enable Gen AI Studio menu
    modelAsService: true         # Enable MaaS dashboard pages
    disableModelServing: false   # Keep traditional model serving enabled
----

Save and exit the editor.

=== Monitor dashboard restart

The dashboard pod needs to restart to pick up the new configuration:

[source,role="execute"]
----
oc get pods -n redhat-ods-applications | grep rhods-dashboard
----

You may see the pod restarting:

----
rhods-dashboard-xxx   0/1     Terminating        0          10m
rhods-dashboard-yyy   0/1     ContainerCreating  0          5s
----

Wait for the new pod to reach Running state:

[source,role="execute"]
----
oc wait --for=condition=ready pod -l app=rhods-dashboard -n redhat-ods-applications --timeout=300s
----

Expected output:

----
pod/rhods-dashboard-xxx condition met
----

=== Verification

Confirm dashboard configuration:

* OdhDashboardConfig shows `genAiStudio: true` and `modelAsService: true`
* Dashboard pod has restarted and is Running
* Dashboard service is accessible

== Exercise 3: Verify Gateway API components

With MaaS enabled, verify that Gateway API resources are configured correctly for model inference routing.

Gateway API provides the ingress layer for MaaS, exposing a single endpoint for all models.

=== Check Gateway resource

List Gateways in the redhat-ods-applications namespace:

[source,role="execute"]
----
oc get gateway -n redhat-ods-applications
----

Expected output:

----
NAME                   CLASS   ADDRESS                                          PROGRAMMED   AGE
maas-default-gateway   istio   maas-default-gateway.apps.{cluster_domain}      True         10m
----

The `PROGRAMMED=True` status indicates the Gateway is configured correctly.

Inspect the Gateway details:

[source,role="execute"]
----
oc describe gateway maas-default-gateway -n redhat-ods-applications
----

Look for:

* `Status: Programmed: True` - Gateway is ready to accept traffic
* `Listeners` section showing HTTP/HTTPS configuration
* `Addresses` showing the external hostname

=== Check HTTPRoute resources

HTTPRoutes define path-based routing rules for the Gateway:

[source,role="execute"]
----
oc get httproute -n redhat-ods-applications
----

Expected output:

----
NAME                  HOSTNAMES                        AGE
maas-api-route        ["maas-default-gateway..."]     10m
----

Describe the HTTPRoute:

[source,role="execute"]
----
oc describe httproute maas-api-route -n redhat-ods-applications
----

Look for:

* `Parent Refs` pointing to maas-default-gateway
* `Rules` section showing `/v1/models` and `/v1/chat/completions` paths
* `Backend Refs` pointing to maas-api service

=== Verify Kuadrant policy resources

Check that Kuadrant policies are configured (these will be populated when governance is configured in Module 4):

[source,role="execute"]
----
oc get authpolicy -n redhat-ods-applications
----

[source,role="execute"]
----
oc get ratelimitpolicy -n redhat-ods-applications
----

These may show "No resources found" initially, which is expected before governance configuration.

=== Verification

Confirm Gateway API setup:

* Gateway resource exists with PROGRAMMED=True status
* Gateway has external address assigned (hostname)
* HTTPRoute exists and references the Gateway
* HTTPRoute defines API paths for model inference
* Kuadrant components (authorino, limitador) are running

== Exercise 4: Access the OpenShift AI dashboard

Now that MaaS is enabled and the dashboard is configured, access the web console to verify the Gen AI Studio interface is available.

The dashboard provides the self-service interface for discovering models, generating API tokens, and viewing usage metrics.

=== Get the dashboard URL

Find the OpenShift AI dashboard route:

[source,role="execute"]
----
oc get route rhods-dashboard -n redhat-ods-applications -o jsonpath='{.spec.host}'
----

Expected output:

----
rhods-dashboard-redhat-ods-applications.apps.{cluster_domain}
----

Or use the attribute placeholder directly: {rhoai_dashboard_url}

=== Log into the dashboard

. Open a web browser (or use the Showroom "OpenShift AI" tab)
. Navigate to the dashboard URL
. Log in with your OpenShift credentials:
  * Username: {user}
  * Password: {password}

You should see the OpenShift AI dashboard home page.

=== Navigate to Gen AI Studio

After logging in:

. Look for "Gen AI Studio" in the left navigation menu

== Troubleshooting

**Issue**: MaaS API pod stuck in ContainerCreating

**Solution**:

. Check pod events: `oc describe pod -l app=maas-api -n redhat-ods-applications`
. Common causes:
  * Image pull issues - verify network connectivity to quay.io
  * Resource constraints - check node capacity
  * PVC binding issues - verify storage class availability
. Check operator logs for reconciliation errors: `oc logs -n redhat-ods-operator deployment/rhods-operator`

**Issue**: Gateway shows PROGRAMMED=False

**Solution**:

. Describe Gateway for error messages: `oc describe gateway maas-default-gateway -n redhat-ods-applications`
. Verify Service Mesh control plane is healthy: `oc get smcp -n istio-system`
. Check Istio operator logs: `oc logs -n openshift-operators deployment/istio-operator`
. Common cause: Service Mesh not installed or control plane not ready

**Issue**: Gen AI Studio menu item not visible after enabling

**Solution**:

. Verify OdhDashboardConfig was updated: `oc get odhdashboardconfig odh-dashboard-config -o jsonpath='{.spec.dashboardConfig.genAiStudio}'` should return `true`
. Force dashboard pod restart: `oc delete pod -l app=rhods-dashboard -n redhat-ods-applications`
. Clear browser cache and refresh dashboard
. Check dashboard pod logs for configuration errors: `oc logs -l app=rhods-dashboard -n redhat-ods-applications`

**Issue**: Dashboard returns 503 Service Unavailable

**Solution**:

. Verify dashboard pod is Running: `oc get pods -l app=rhods-dashboard -n redhat-ods-applications`
. Check dashboard service exists: `oc get svc rhods-dashboard -n redhat-ods-applications`
. Verify route is configured: `oc get route rhods-dashboard -n redhat-ods-applications`
. Check for NetworkPolicy blocking traffic

**Issue**: Kuadrant pods not deploying after MaaS enablement

**Solution**:

. Check DataScienceCluster status for errors: `oc get datasciencecluster default-dsc -o yaml | grep -A 10 conditions`
. Verify Kuadrant operator is installed: `oc get csv -n openshift-operators | grep kuadrant`
. If Kuadrant operator is missing, it should be installed automatically by OpenShift AI operator
. Check OpenShift AI operator logs: `oc logs -n redhat-ods-operator deployment/rhods-operator | grep -i kuadrant`

**Issue**: HTTPRoute not routing traffic to MaaS API

**Solution**:

. Verify HTTPRoute exists and is attached to Gateway: `oc get httproute -n redhat-ods-applications`
. Check HTTPRoute status: `oc describe httproute maas-api-route -n redhat-ods-applications`
. Verify backend service is healthy: `oc get svc maas-api -n redhat-ods-applications`
. Test service directly: `oc port-forward svc/maas-api 8080:8080 -n redhat-ods-applications` then `curl localhost:8080/v1/models`
