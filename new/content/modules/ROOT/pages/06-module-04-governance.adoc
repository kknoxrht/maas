== Exercise 1: Understanding MaaS governance tiers

Before configuring governance, you need to understand how tier-based access control works and why it's critical for multi-tenant model serving.

Without tiers, centralized serving becomes uncontrolled consumption. Tier-based access transforms the MaaS platform from an unlimited resource drain to fair allocation across departments.

=== The governance challenge at ChocoTech

ChocoTech controls $500,000 worth of GPU infrastructure serving four departments:

* **Quality Control**: Defect detection (critical, production workload)
* **R&D**: Recipe optimization (important, but not real-time)
* **Marketing**: Sentiment analysis (high volume, non-critical)
* **Supply Chain**: Inventory prediction (batch processing)

**Without governance**:

* Marketing runs unlimited chatbot queries, exhausting GPU capacity
* Quality Control's critical defect detection gets rate-limited during production shifts
* Finance cannot allocate AI costs accurately (no consumption tracking)
* Security has no audit trail showing who accessed which models

**With MaaS governance**:

* Each department is assigned a tier with appropriate rate limits and quotas
* Quality Control gets "Platinum" tier with high limits and priority access
* Marketing gets "Bronze" tier with moderate limits appropriate for their use case
* Finance can perform chargeback based on actual token consumption by department
* Security has complete audit logs of every API request with user identity

=== What are MaaS tiers?

Tiers are access control layers that define:

**Access level**::
Which models a user can query (Bronze tier can't access Platinum-only models)

**Rate limits**::
Maximum requests per minute (Bronze: 100 req/min, Platinum: 10,000 req/min)

**Token quotas**::
Maximum tokens consumed per minute (Bronze: 10,000 tokens/min, Platinum: 1,000,000 tokens/min)

**Group mapping**::
Which OpenShift groups are assigned to each tier

**Priority level**::
Determines access hierarchy when users belong to multiple groups across different tiers

Example: A developer with a Bronze tier Bearer token can only access models published to "Bronze tier and above." A Platinum user can access all models.

=== Default tier structure for ChocoTech

You'll configure four tiers aligned with ChocoTech's business priorities:

[cols="1,1,2,2"]
|===
|Tier Name |Priority Level |Target Department |Rate Limit (req/min)

|Bronze
|Level 0
|Marketing (exploratory)
|100 requests/min, 10K tokens/min

|Silver
|Level 1
|Supply Chain (batch)
|500 requests/min, 50K tokens/min

|Gold
|Level 2
|R&D (development)
|2,000 requests/min, 200K tokens/min

|Platinum
|Level 3
|Quality Control (production)
|10,000 requests/min, 1M tokens/min
|===

Priority Level determines access hierarchy: when a user belongs to multiple groups across different tiers, MaaS assigns them to the tier with the highest level number.

=== Group assignment logic

MaaS uses OpenShift group membership to assign users to tiers:

. User authenticates with Bearer token
. Authorino (Kuadrant authentication component) extracts user identity and group memberships from OpenShift OAuth
. MaaS evaluates which tier configurations match the user's groups
. Highest priority level wins if the user belongs to multiple tiers
. Access is enforced based on assigned tier and model configuration
. Limitador (Kuadrant rate limiting component) tracks consumption and enforces quotas

A Platform Engineer creates a group `chocotech-quality-control` in OpenShift, then maps it to the Platinum tier. All members of that group automatically get Platinum access with highest rate limits.

== Exercise 3: Configure governance tiers and rate limiting

Now you'll create MaaSTier custom resources that map OpenShift groups to tier levels with rate limits and quotas.

Tier configuration is declarative, using custom resources managed by the MaaS operator.

=== Create the Platinum tier (Quality Control)

Create a manifest for the highest tier:

[source,role="execute"]
----
cat > tier-platinum.yaml <<EOF
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSTier
metadata:
  name: platinum
  namespace: redhat-ods-applications
spec:
  priorityLevel: 3
  groups:
    - chocotech-quality-control
  rateLimit:
    requestsPerMinute: 10000
    tokensPerMinute: 1000000
  quotas:
    dailyTokenLimit: 100000000
  description: "Production workloads - Quality Control team defect detection"
EOF
----

Apply the tier configuration:

[source,role="execute"]
----
oc apply -f tier-platinum.yaml
----

== Exercise 4: Deploy Grafana for observability

ChocoTech's Finance department needs visibility into model usage for chargeback. You'll deploy Grafana dashboards to monitor token consumption and GPU utilization.

Grafana provides the observability layer for tracking model usage across user tiers and different models.

=== Install Grafana operator (if not already installed)

Check if Grafana operator is installed:

[source,role="execute"]
----
oc get csv -n openshift-operators | grep grafana
----

If not installed, install the Grafana operator from OperatorHub:

[source,role="execute"]
----
cat > grafana-operator-subscription.yaml <<EOF
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: grafana-operator
  namespace: openshift-operators
spec:
  channel: v5
  name: grafana-operator
  source: community-operators
  sourceNamespace: openshift-marketplace
EOF
----

[source,role="execute"]
----
oc apply -f grafana-operator-subscription.yaml
----

Wait for the operator to be ready:

[source,role="execute"]
----
oc wait --for=condition=ready pod -l name=grafana-operator -n openshift-operators --timeout=300s
----

=== Create Grafana instance for MaaS monitoring

Create a Grafana instance in the OpenShift AI namespace:

[source,role="execute"]
----
cat > grafana-instance.yaml <<EOF
apiVersion: grafana.integreatly.org/v1beta1
kind: Grafana
metadata:
  name: maas-grafana
  namespace: redhat-ods-applications
  labels:
    dashboards: "maas"
spec:
  config:
    security:
      admin_user: admin
      admin_password: chocotech123
    server:
      root_url: https://maas-grafana.apps.{cluster_domain}
EOF
----

[source,role="execute"]
----
oc apply -f grafana-instance.yaml
----

Wait for Grafana to deploy:

[source,role="execute"]
----
oc wait --for=condition=ready pod -l app=grafana -n redhat-ods-applications --timeout=300s
----

=== Create Grafana data source for Prometheus

Configure Grafana to query OpenShift's Prometheus for metrics:

[source,role="execute"]
----
cat > grafana-datasource.yaml <<EOF
apiVersion: grafana.integreatly.org/v1beta1
kind: GrafanaDataSource
metadata:
  name: prometheus-maas
  namespace: redhat-ods-applications
spec:
  instanceSelector:
    matchLabels:
      dashboards: "maas"
  datasource:
    name: Prometheus
    type: prometheus
    access: proxy
    url: https://thanos-querier.openshift-monitoring.svc.cluster.local:9091
    isDefault: true
    jsonData:
      tlsSkipVerify: true
      timeInterval: "5s"
EOF
----

[source,role="execute"]
----
oc apply -f grafana-datasource.yaml
----

== Exercise 5: Generate chargeback reports

Now that governance and observability are configured, you can generate chargeback reports for Finance showing token consumption by department.

Token-based consumption tracking enables Finance to bill departments based on actual usage rather than infrastructure allocation.

=== Query token consumption metrics

Use Prometheus to query token consumption by tier:

[source,role="execute"]
----
oc exec -n openshift-monitoring prometheus-k8s-0 -- promtool query instant \
  http://localhost:9090 \
  'sum(rate(maas_tokens_consumed_total[1h])) by (tier, model)'
----

This returns token consumption rate by tier and model over the last hour.

Example output:

----
tier="platinum", model="llama-3-8b"    125000 tokens/hour
tier="gold", model="llama-3-8b"         45000 tokens/hour
tier="bronze", model="nemotron-8b"      12000 tokens/hour
----

=== Calculate chargeback costs

Define cost per 1,000 tokens for internal chargeback:

* Bronze tier: $0.001 per 1,000 tokens
* Silver tier: $0.002 per 1,000 tokens
* Gold tier: $0.003 per 1,000 tokens
* Platinum tier: $0.005 per 1,000 tokens

Example calculation for Quality Control (Platinum tier):

----
Tokens consumed (24 hours): 125,000 tokens/hour x 24 hours = 3,000,000 tokens
Cost: (3,000,000 / 1,000) x $0.005 = $15.00 per day
Monthly estimate: $15.00 x 30 = $450.00
----

Marketing (Bronze tier):

----
Tokens consumed (24 hours): 12,000 tokens/hour x 24 hours = 288,000 tokens
Cost: (288,000 / 1,000) x $0.001 = $0.29 per day
Monthly estimate: $0.29 x 30 = $8.70
----

This demonstrates that Quality Control's production workload justifies their Platinum tier allocation, while Marketing's exploratory use appropriately stays in Bronze tier.

=== Create a chargeback report

Generate a simple chargeback report:

[source,role="execute"]
----
cat > chargeback-report.sh <<'EOF'
#!/bin/bash
echo "ChocoTech Industries - MaaS Chargeback Report"
echo "Generated: $(date)"
echo ""
echo "Department          Tier      Tokens (24h)    Cost/Day    Monthly Est."
echo "--------------------------------------------------------------------------------"
echo "Quality Control     Platinum  3,000,000       \$15.00     \$450.00"
echo "R&D                 Gold      1,080,000       \$3.24      \$97.20"
echo "Supply Chain        Silver      480,000       \$0.96      \$28.80"
echo "Marketing           Bronze      288,000       \$0.29      \$8.70"
echo "--------------------------------------------------------------------------------"
echo "Total Monthly Cost:                                     \$584.70"
EOF
chmod +x chargeback-report.sh
----

Run the report:

[source,role="execute"]
----
./chargeback-report.sh
----

This simple report demonstrates the concept. In production, you would integrate Prometheus metrics with your organization's billing system or create automated Grafana reports.

== Troubleshooting

**Issue**: RateLimitPolicy not enforcing quotas (requests succeeding beyond limits)

**Solution**:

. Verify Limitador pod is healthy: `oc get pods -n kuadrant-system | grep limitador`
. Check RateLimitPolicy is correctly configured: `oc describe ratelimitpolicy -n redhat-ods-applications`
. Verify AuthPolicy is configured to extract user tier: `oc get authpolicy -n redhat-ods-applications`
. Check Limitador logs for policy evaluation: `oc logs -l app=limitador -n kuadrant-system`

**Issue**: User assigned to wrong tier (expected Platinum, getting Bronze)

**Solution**:

. Verify user group membership: `oc get user {user} -o yaml | grep -A 5 groups`
. Check MaaSTier configuration: `oc get maastier platinum -o yaml`
. Ensure group name matches exactly in tier spec
. Priority level determines tier when user belongs to multiple groups (highest wins)
. Check Authorino logs for user tier resolution: `oc logs -l app=authorino -n kuadrant-system`

**Issue**: Grafana cannot connect to Prometheus data source

**Solution**:

. Verify Prometheus is accessible from Grafana namespace
. Check GrafanaDataSource status: `oc describe grafanadatasource prometheus-maas -n redhat-ods-applications`
. Verify Thanos Querier service exists: `oc get svc thanos-querier -n openshift-monitoring`
. Test connectivity: `oc exec -n redhat-ods-applications deployment/maas-grafana -- curl -k https://thanos-querier.openshift-monitoring.svc.cluster.local:9091/api/v1/query?query=up`

**Issue**: Token consumption metrics not appearing in Prometheus

**Solution**:

. Verify KServe and Kuadrant are exposing metrics
. Check ServiceMonitor resources: `oc get servicemonitor -n redhat-ods-applications`
. Verify Prometheus is scraping MaaS metrics: Query `up{job="maas-api"}` in Prometheus UI
. Check MaaS API pod logs for metric export: `oc logs -l app=maas-api -n redhat-ods-applications | grep metrics`

**Issue**: MaaSTier resource created but RateLimitPolicy not generated

**Solution**:

. Check MaaS operator logs: `oc logs -n redhat-ods-operator deployment/rhods-operator | grep -i maastier`
. Verify Kuadrant CRDs are installed: `oc get crd | grep kuadrant`
. Check MaaSTier status for reconciliation errors: `oc get maastier platinum -o yaml | grep -A 10 status`
. Verify RBAC permissions for MaaS operator to create RateLimitPolicy resources
