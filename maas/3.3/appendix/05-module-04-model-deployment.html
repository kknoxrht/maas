Exercise 5: Deploy a second model for multi-model catalog

ChocoTech’s Marketing team requests access to a smaller, faster model for sentiment analysis. Deploy a second model to demonstrate multi-model catalog management.

This exercise shows how easily you can expand the model catalog as new business requirements emerge.

Deploy Nemotron 8B for Marketing team

Create a manifest for a second model:

cat > nemotron-8b-llminferenceservice.yaml <<EOF
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: nemotron-8b
  namespace: chocotech-models
  labels:
    app: nemotron-8b
    team: marketing
spec:
  model:
    name: nvidia/nemotron-4-8b-instruct
    source: huggingface
  runtime:
    name: vllm
    version: v0.5.4
  resources:
    limits:
      nvidia.com/gpu: 1
    requests:
      nvidia.com/gpu: 1
      memory: 16Gi
  scaling:
    minReplicas: 1
    maxReplicas: 2
  maasConfig:
    publishToGenAIStudio: true
    tierAccess:
      - bronze
      - silver
EOF

Note: This model is restricted to bronze and silver tiers, demonstrating selective access control.

Apply the manifest:

oc apply -f nemotron-8b-llminferenceservice.yaml

Wait for deployment:

oc wait --for=condition=ready llminferenceservice/nemotron-8b -n chocotech-models --timeout=600s

Expected output:

llminferenceservice.serving.kserve.io/nemotron-8b condition met

Verify multi-model catalog

List all deployed models:

oc get llminferenceservice -n chocotech-models

Expected output:

NAME           READY   REASON   AGE
llama-3-8b     True    Ready    20m
nemotron-8b    True    Ready    5m

Check the model catalog via API:

curl https://$(oc get route maas-default-gateway -n redhat-ods-applications -o jsonpath='{.spec.host}')/v1/models

Expected output showing both models:

{
  "object": "list",
  "data": [
    {
      "id": "meta-llama/Meta-Llama-3-8B-Instruct",
      "object": "model",
      "owned_by": "chocotech-models"
    },
    {
      "id": "nvidia/nemotron-4-8b-instruct",
      "object": "model",
      "owned_by": "chocotech-models"
    }
  ]
}

Refresh the Gen AI Studio Models tab in the dashboard. You should now see both models listed with their respective tier access controls.

Verification

Confirm multi-model catalog:

  • Both LLMInferenceServices show READY=True status

  • Both models appear in Gen AI Studio catalog

  • Models show different tier access (Llama-3: all tiers, Nemotron: bronze/silver only)

  • /v1/models API endpoint returns both models

  • Each model has its own service and pods in chocotech-models namespace

Exercise 6: Production lifecycle management and GitOps

ChocoTech’s Quality Control team has been successfully using Llama-3 8B for defect analysis for several weeks. Now they want to upgrade to a new version that has been fine-tuned on chocolate manufacturing defects.

"We can’t risk downtime," explains the QC Director. "Our production line depends on real-time defect detection. Can we test the new model with 10% of traffic first, then gradually roll it out if it performs well?"

This is a classic production deployment challenge: how do you safely update AI models without disrupting business-critical workloads?

In this exercise, you’ll learn production-grade model lifecycle management including version management, canary deployments, GitOps integration with Argo CD, and rollback procedures.

Version management strategy

Production AI systems require disciplined version management for models, just like application code.

Semantic versioning for models

Use semantic versioning to track model releases:

  • v1.0.0 - Initial production release

  • v1.1.0 - Minor improvement (fine-tuning, hyperparameter optimization)

  • v2.0.0 - Major change (architecture change, new training data)

ChocoTech versioning example:

Model Storage Path Structure:
s3://chocotech-models/
├── llama3-8b-defect-analysis/
│   ├── v1.0.0/  (Base Llama-3 8B Instruct)
│   ├── v1.1.0/  (Fine-tuned on chocolate defect images - 500 examples)
│   └── v2.0.0/  (Fine-tuned on chocolate defect images - 5000 examples + multimodal)

Deploying multiple versions simultaneously

Create separate LLMInferenceService resources for each version:

apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama3-defect-v1
  namespace: chocotech-models
  labels:
    app.kubernetes.io/name: llama3-defect-analysis
    app.kubernetes.io/version: "1.0.0"
spec:
  model:
    name: llama3-8b-defect-analysis
    source: s3
    storage:
      path: s3://chocotech-models/llama3-8b-defect-analysis/v1.0.0/
      key: model-storage-connection
  runtime:
    name: vllm
    version: v0.5.4
  resources:
    limits:
      nvidia.com/gpu: 1
  scaling:
    minReplicas: 2
    maxReplicas: 5
  maasConfig:
    publishToGenAIStudio: true
    tierAccess:
      - gold
      - platinum

Notice the app.kubernetes.io/version label tracks the semantic version, while the resource name includes the version (llama3-defect-v1).

Canary deployments with traffic splitting

Canary deployments gradually shift traffic from the old version to the new version, allowing you to monitor performance and catch issues before full rollout.

Traffic splitting with Gateway API

OpenShift AI MaaS uses Gateway API HTTPRoute resources for traffic management. Create a canary route that splits traffic between v1 (90%) and v2 (10%):

cat > llama3-defect-canary-route.yaml <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: llama3-defect-canary
  namespace: chocotech-models
spec:
  parentRefs:
    - name: maas-default-gateway
      namespace: redhat-ods-applications
  hostnames:
    - "llama3-defect.apps.{cluster_domain}"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /v1/
      backendRefs:
        - name: llama3-defect-v1-predictor-00001
          port: 8080
          weight: 90
        - name: llama3-defect-v2-predictor-00001
          port: 8080
          weight: 10
EOF
oc apply -f llama3-defect-canary-route.yaml

How canary traffic splitting works:

Incoming Requests (100%)
         │
         ▼
    MaaS Gateway
         │
    ┌────┴────┐
    │         │
   90%       10%
    │         │
    ▼         ▼
  v1.0.0    v2.0.0
(Production) (Canary)

90% of requests go to the stable v1 deployment, while 10% are routed to the new v2 canary version.

Monitoring canary performance

During the canary period, monitor key metrics:

Success rate:

# Count successful requests for v1
oc logs -l app.kubernetes.io/version=1.0.0 -n chocotech-models | grep "HTTP/1.1 200" | wc -l

# Count successful requests for v2
oc logs -l app.kubernetes.io/version=2.0.0 -n chocotech-models | grep "HTTP/1.1 200" | wc -l

Error rate:

# Count errors for v2 canary
oc logs -l app.kubernetes.io/version=2.0.0 -n chocotech-models | grep -E "HTTP/1.1 [45][0-9]{2}" | wc -l

Latency comparison:

Check average response times in model logs or use Prometheus metrics (covered in Module 4):

# Example PromQL query for latency comparison
histogram_quantile(0.95,
  rate(vllm_llmd_request_duration_seconds_bucket{version="2.0.0"}[5m])
)

ChocoTech canary success criteria:

  • Error rate < 2% for v2 canary

  • P95 latency < 500ms for v2 canary

  • Quality Control team manually validates defect detection accuracy on test images

  • Monitor for 2 hours before promotion decision

Promoting canary to production

If canary metrics meet success criteria, gradually increase traffic to v2:

Increase canary to 50%:

oc patch httproute llama3-defect-canary -n chocotech-models --type=merge -p '
{
  "spec": {
    "rules": [
      {
        "matches": [{"path": {"type": "PathPrefix", "value": "/v1/"}}],
        "backendRefs": [
          {"name": "llama3-defect-v1-predictor-00001", "port": 8080, "weight": 50},
          {"name": "llama3-defect-v2-predictor-00001", "port": 8080, "weight": 50}
        ]
      }
    ]
  }
}'

After monitoring 50/50 split for 1 hour, promote to 100% v2:

oc patch httproute llama3-defect-canary -n chocotech-models --type=merge -p '
{
  "spec": {
    "rules": [
      {
        "matches": [{"path": {"type": "PathPrefix", "value": "/v1/"}}],
        "backendRefs": [
          {"name": "llama3-defect-v2-predictor-00001", "port": 8080, "weight": 100}
        ]
      }
    ]
  }
}'

Once v2 is fully promoted and stable, you can safely delete the v1 deployment:

oc delete llminferenceservice llama3-defect-v1 -n chocotech-models

Rolling back a failed canary

If the canary version shows problems (high error rate, poor quality, latency spikes), immediately roll back to 100% v1:

oc patch httproute llama3-defect-canary -n chocotech-models --type=merge -p '
{
  "spec": {
    "rules": [
      {
        "matches": [{"path": {"type": "PathPrefix", "value": "/v1/"}}],
        "backendRefs": [
          {"name": "llama3-defect-v1-predictor-00001", "port": 8080, "weight": 100}
        ]
      }
    ]
  }
}'

Then investigate the v2 deployment issue before attempting another canary.

For production environments, use GitOps to manage model deployments declaratively from a Git repository.

Environment-specific overlays

Production environment overlay with Kustomize:

# environments/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: chocotech-models-prod

resources:
  - ../../base/llminferenceservice-template.yaml

patches:
  - target:
      kind: LLMInferenceService
    patch: |-
      - op: replace
        path: /metadata/name
        value: llama3-defect-v2
      - op: replace
        path: /metadata/labels/app.kubernetes.io~1version
        value: "2.0.0"
      - op: replace
        path: /spec/model/name
        value: meta-llama/Meta-Llama-3-8B-Instruct
      - op: replace
        path: /spec/resources/limits/nvidia.com~1gpu
        value: 1
      - op: replace
        path: /spec/scaling/minReplicas
        value: 3
      - op: replace
        path: /spec/scaling/maxReplicas
        value: 10
      - op: replace
        path: /spec/maasConfig/publishToGenAIStudio
        value: true

Argo CD Application for automated sync

Create an Argo CD Application to continuously sync the Git repository to the cluster:

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: chocotech-models-production
  namespace: openshift-gitops
spec:
  project: default
  source:
    repoURL: https://github.com/chocotech/maas-models.git
    targetRevision: main
    path: environments/production
  destination:
    server: https://kubernetes.default.svc
    namespace: chocotech-models-prod
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m

How GitOps workflow works for ChocoTech:

1. Data Scientist commits new model version to Git
   ├─> Git: environments/production/kustomization.yaml (updated model path)

2. Argo CD detects change
   ├─> Compares Git state vs cluster state
   ├─> Identifies drift: model version v1.0.0 → v2.0.0

3. Argo CD syncs automatically
   ├─> Applies Kustomize overlay
   ├─> Updates LLMInferenceService in cluster

4. OpenShift AI operator reconciles
   ├─> Downloads new model version
   ├─> Performs rolling update
   ├─> Updates MaaS catalog

5. Platform Engineer monitors sync status
   ├─> Argo CD dashboard shows sync health
   ├─> Model deployment progresses: Downloading → Loading → Ready

Apply the Argo CD application (if Argo CD is installed):

This step requires OpenShift GitOps operator. If not installed, skip this section and use manual oc apply workflow. 
cat > argocd-chocotech-models.yaml <<EOF

Rollback with GitOps

To rollback a bad deployment using GitOps:

Option 1: Git revert (recommended)

cd chocotech-maas-models
git log --oneline
# Identify the commit that introduced the bad version
git revert abc123
git push origin main
# Argo CD automatically syncs the rollback

Option 2: Manual sync to previous commit

In Argo CD UI:

  1. Navigate to chocotech-models-production application

  2. Click "History and Rollback"

  3. Select previous successful deployment

  4. Click "Rollback"

Argo CD reverts the cluster state to match the selected Git commit.

Automated testing in CI/CD pipeline

Integrate model validation into your GitOps pipeline before production deployment:

# .github/workflows/model-validation.yaml
name: Validate Model Deployment

on:
  pull_request:
    branches: [main]
    paths:
      - 'environments/production/**'

jobs:
  validate-model:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Validate Kustomize manifests
        run: |
          kustomize build environments/production > rendered.yaml
          kubectl apply --dry-run=client -f rendered.yaml

      - name: Test model inference (staging environment)
        env:
          STAGING_API_URL: ${{ secrets.STAGING_API_URL }}
          STAGING_API_KEY: ${{ secrets.STAGING_API_KEY }}
        run: |
          # Test that model responds correctly
          response=$(curl -X POST "$STAGING_API_URL/v1/chat/completions" \
            -H "Authorization: Bearer $STAGING_API_KEY" \
            -H "Content-Type: application/json" \
            -d '{
              "model": "meta-llama/Meta-Llama-3-8B-Instruct",
              "messages": [{"role": "user", "content": "Test query"}],
              "max_tokens": 50
            }')

          # Verify response contains expected fields
          echo "$response" | jq -e '.choices[0].message.content' || exit 1

      - name: Performance benchmark
        run: |
          # Run load test to ensure latency requirements
          # Example: 100 requests, P95 latency < 500ms
          ./scripts/benchmark-model.sh --requests 100 --p95-threshold 500

      - name: Comment PR with validation results
        uses: actions/github-script@v6
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '✅ Model validation passed. Safe to merge.'
            })

This CI/CD pipeline ensures that:

  • Kustomize manifests are syntactically valid

  • Model responds correctly to test queries

  • Performance meets latency requirements

  • Pull requests show validation status before merge

Best practices for production lifecycle management

Version management:

  • Always use semantic versioning for model releases

  • Keep at least one previous version deployed for instant rollback

  • Store model artifacts in versioned S3/PVC paths

  • Tag LLMInferenceService resources with version labels

Canary deployments:

  • Start with 10% canary traffic, monitor for at least 1 hour

  • Define success criteria before deployment (error rate, latency, quality metrics)

  • Gradually increase canary traffic: 10% → 25% → 50% → 100%

  • Have rollback plan ready before starting canary

GitOps adoption:

  • Use Git as the single source of truth for all model deployments

  • Separate environments (dev/staging/production) with different configurations

  • Enable automated sync for non-production, require approval for production

  • Use Kustomize overlays for environment-specific configuration

Monitoring and observability:

  • Track model version in all logs and metrics

  • Monitor error rates, latency (TTFT, TPOT), and GPU utilization per version

  • Set up alerts for canary failure conditions

  • Retain logs for at least 30 days for incident investigation

Change management:

  • Test model changes in dev → staging → production progression

  • Require peer review for production model updates

  • Document rollback procedures and test them regularly

  • Schedule model updates during low-traffic periods

Troubleshooting

Issue: Canary traffic is not splitting correctly (all traffic goes to one version)

Solution:

  1. Verify HTTPRoute weights sum to 100: oc get httproute llama3-defect-canary -n chocotech-models -o yaml | grep weight

  2. Check that both backend services exist: oc get svc -n chocotech-models | grep llama3-defect

  3. Verify Gateway API is routing through MaaS gateway: oc describe httproute llama3-defect-canary -n chocotech-models

  4. Test with curl and observe response headers to confirm version distribution

Issue: Argo CD Application stuck in "OutOfSync" state

Solution:

  1. Check Argo CD application health: oc get application chocotech-models-production -n openshift-gitops -o yaml

  2. Common causes:

    • Kustomize build errors: Run kustomize build environments/production locally to validate

    • Namespace doesn’t exist: Argo CD should create it, but verify with oc get namespace chocotech-models-prod

    • Resource conflicts: Check if resources already exist from manual deployment

  3. Force sync in Argo CD UI or via CLI: argocd app sync chocotech-models-production --force

Issue: Model rollback via Git revert doesn’t trigger re-deployment

Solution:

  1. Verify Argo CD automated sync is enabled: oc get application chocotech-models-production -n openshift-gitops -o jsonpath='{.spec.syncPolicy.automated}'

  2. Check Argo CD is monitoring the correct branch: oc get application chocotech-models-production -n openshift-gitops -o jsonpath='{.spec.source.targetRevision}'

  3. Manually trigger sync if needed: argocd app sync chocotech-models-production

  4. Verify Git commit was pushed to remote: git log origin/main

Issue: New model version downloads successfully but fails to load with CUDA errors

Solution:

  1. Check GPU availability on node: oc describe node -l nvidia.com/gpu.present=true | grep "Allocated resources" -A 10

  2. Verify vLLM version compatibility with model: Some models require specific vLLM versions

  3. Check model logs for detailed error: oc logs -l app.kubernetes.io/version=2.0.0 -n chocotech-models --tail=100

  4. Common causes: insufficient VRAM, incompatible quantization, model architecture not supported by vLLM

Issue: CI/CD pipeline fails model validation even though model works locally

Solution:

  1. Verify staging environment has sufficient resources: Model may be resource-constrained in staging

  2. Check API key is valid and has correct tier access: Expired or wrong tier tokens cause 401/403 errors

  3. Validate test query is appropriate for model: Complex queries may timeout in validation pipeline

  4. Check network connectivity from CI/CD runner to staging cluster

  5. Review GitHub Actions logs for specific error messages

Troubleshooting

Issue: LLMInferenceService stuck in "DownloadingModel" phase for >10 minutes

Solution:

  1. Check pod events for image pull errors: oc describe pod -l app=llama-3-8b -n chocotech-models

  2. Common causes:

    • Hugging Face model download timeout (large models take time)

    • Network connectivity issues to models.huggingface.co

    • Insufficient storage on GPU node for model caching

  3. Check pod logs: oc logs -l app=llama-3-8b -n chocotech-models

Issue: Model pod fails with "Out of memory" error

Solution:

  1. Check GPU memory availability: oc describe node -l nvidia.com/gpu.present=true | grep -A 5 "Allocatable:"

  2. Verify model size fits on GPU (Llama-3 8B requires ~16GB, 70B requires 140GB+)

  3. Adjust resource requests in LLMInferenceService spec

  4. For large models, ensure distributed inference is configured with sufficient worker replicas

Issue: Model doesn’t appear in Gen AI Studio catalog

Solution:

  1. Verify publishToGenAIStudio: true in LLMInferenceService spec

  2. Check namespace has modelsAsService=true label: oc get namespace chocotech-models --show-labels

  3. Verify MaaSModel resource was created: oc get maasmodel -n chocotech-models

  4. Check MaaS API pod logs: oc logs -l app=maas-api -n redhat-ods-applications

  5. Refresh browser cache and reload Gen AI Studio dashboard

Issue: vLLM serving runtime fails to start with CUDA errors

Solution:

  1. Verify GPU drivers are loaded on node: oc debug node/<node-name> — chroot /host nvidia-smi

  2. Check NVIDIA GPU operator pods: oc get pods -n nvidia-gpu-operator

  3. Verify GPU node has correct labels: oc get nodes -l nvidia.com/gpu.present=true --show-labels

  4. Check vLLM version compatibility with CUDA driver version

Issue: LeaderWorkerSet not creating worker pods

Solution:

  1. Verify LeaderWorkerSet operator is running: oc get pods -n leaderworkerset-system

  2. Check LLMInferenceService for distributed inference configuration

  3. Describe LeaderWorkerSet resource: oc get leaderworkerset -n chocotech-models

  4. Review llm-serving-runtime controller logs: oc logs -l app=llm-serving-runtime-controller -n redhat-ods-applications

Issue: Model returns 500 Internal Server Error after deployment

Solution:

  1. Check model pod health: oc get pods -l app=llama-3-8b -n chocotech-models

  2. Review model logs for runtime errors: oc logs -l app=llama-3-8b -n chocotech-models --tail=100

  3. Verify model inference endpoint: oc port-forward svc/llama-3-8b-predictor-00001 8080:8080 -n chocotech-models then curl localhost:8080/v1/models

  4. Common causes: incompatible model format, insufficient GPU memory, vLLM configuration errors
