2.3: Deploying governance tier policies and verifying readiness

With organizational structure and authentication in place, the final step is to deploy tier-based governance policies that enforce rate limits and quotas across ChocoTech’s business units.

This section covers creating MaaSTier custom resources, verifying governance components, and completing a comprehensive pre-deployment readiness checklist.

Understanding MaaS tier governance

Tier governance ensures fair resource allocation and prevents any single business unit from monopolizing GPU resources.

What are MaaSTier custom resources?

MaaSTier is a custom resource definition (CRD) provided by the Red Hat OpenShift AI MaaS operator. Each MaaSTier resource defines:

  • Tier properties: Display name, priority level

  • Rate limits: Maximum requests per minute

  • Quotas: Maximum tokens per hour for LLM inference

  • Group mapping: Which OpenShift groups belong to this tier

When you deploy a model in Module 3, you’ll annotate it with tier names. The MaaS platform enforces the tier’s rate limits and quotas on all requests to that model.

Why tier policies matter

Without tier governance:

  • Bronze tier users (Marketing’s exploratory experiments) could consume all available GPU capacity

  • Gold tier users (Quality Control’s production defect detection) would be blocked waiting for resources

  • No mechanism exists to track or limit costs per business unit

With tier governance:

  • Prioritized request processing: Gold tier requests are processed first during peak demand

  • Fair resource allocation: Rate limits prevent Bronze tier from consuming all capacity

  • Cost control: Quotas cap token consumption (and thus GPU costs) per tier

  • Chargeback reporting: Track AI infrastructure costs by tier and business unit

The tier enforcement flow

When an application makes an inference request, the request flows through multiple governance layers:

API Request (with Bearer token)
  ↓
Gateway (maas-default-gateway)
  ↓
Authorino (validates token, extracts ServiceAccount identity)
  ↓
Authorino (determines which group/tier the ServiceAccount belongs to)
  ↓
Limitador (checks tier rate limit: has this tier exceeded requests/minute?)
  ↓ (if rate limit OK)
MaaS API (checks tier quota: has this tier exceeded tokens/hour?)
  ↓ (if quota OK)
Model (LLMInferenceService processes the request)

If rate limit is exceeded: HTTP 429 Too Many Requests If quota is exceeded: HTTP 429 with quota exhaustion message

Creating MaaSTier custom resources

Deploy tier policies for Gold, Silver, and Bronze tiers.

Exercise 1: Deploy Gold tier policy

Create the Gold tier policy for Quality Control:

oc apply -f - <<EOF
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSTier
metadata:
  name: gold
spec:
  displayName: "Gold Tier - Production Workloads"
  priority: 100
  rateLimit:
    requestsPerMinute: 100
  quota:
    tokensPerHour: 50000
  groups:
  - chocotech-quality-control
EOF

Expected output:

maastier.maas.opendatahub.io/gold created

Understanding the Gold tier definition

  • priority: 100: Highest priority level for request processing

    • During peak demand, Gold tier requests are processed before Silver and Bronze

    • Ensures Quality Control’s production defect detection system gets guaranteed resources

  • rateLimit.requestsPerMinute: 100: Maximum 100 inference requests per minute

    • Prevents excessive burst traffic that could overwhelm GPU resources

    • Allows steady production throughput

  • quota.tokensPerHour: 50000: Maximum 50,000 LLM tokens per hour

    • Tokens = input tokens + output tokens for all requests in the past hour

    • For context: GPT-3.5 level model processes ~500 tokens per request

    • 50,000 tokens ≈ 100 requests of 500 tokens each

  • groups: [chocotech-quality-control]: Maps this tier to the OpenShift group

    • Members of chocotech-quality-control group automatically belong to Gold tier

    • When they generate tokens from ServiceAccounts, requests count against Gold tier limits

The Silver tier provides balanced resources:

  • Half the rate limit of Gold tier (50 vs 100 requests/minute)

  • Half the quota of Gold tier (25,000 vs 50,000 tokens/hour)

  • Medium priority (50) - processed after Gold, before Bronze

This allocation is appropriate for R&D’s recipe optimization work, which is important but not production-critical.

Configuring Authorino authentication policies

Authorino validates JWT tokens and enforces authentication policies. Verify the MaaS operator created the necessary AuthConfig resources.

Exercise 5: Verify Authorino AuthConfig

The MaaS operator should have created an AuthConfig for token authentication when you enabled MaaS in Module 1. Verify it exists:

oc get authconfig -n redhat-ods-applications

Expected output:

NAME        AGE
maas-auth   15m

Describe the AuthConfig to see its configuration:

oc describe authconfig maas-auth -n redhat-ods-applications

Expected output (partial):

Name:         maas-auth
Namespace:    redhat-ods-applications
Spec:
  Hosts:
    maas-default-gateway-redhat-ods-applications.apps.cluster.example.com
  Authentication:
    k8s-service-account-tokens:
      Kubernetes Token Review:
        Audiences:
          https://kubernetes.default.svc
  Authorization:
    k8s-rbac:
      Kubernetes Subject Access Review:
        User:
          Selector:  auth.identity.user.username

Understanding the AuthConfig

This AuthConfig provides:

  • Host binding: Applies to requests sent to the MaaS Gateway hostname

  • Token validation: Uses Kubernetes TokenReview API to validate JWT signatures

    • Ensures tokens are signed by the Kubernetes API server

    • Rejects tampered or forged tokens

  • ServiceAccount extraction: Extracts ServiceAccount identity from validated tokens

  • RBAC authorization: Checks if the ServiceAccount has required permissions

    • Uses Kubernetes SubjectAccessReview API

    • Enforces the RBAC Roles and RoleBindings you created in Section 2.2

What if AuthConfig is missing?

If the oc get authconfig command returns "No resources found", the MaaS operator may not have created it automatically. Recreate it manually:

In most cases, the AuthConfig should exist. Only recreate if it’s genuinely missing.

 
# Only run this if AuthConfig does not exist
oc apply -f - <<EOF
apiVersion: authorino.kuadrant.io/v1beta2
kind: AuthConfig
metadata:
  name: maas-auth
  namespace: redhat-ods-applications
spec:
  hosts:
  - maas-default-gateway-redhat-ods-applications.apps.cluster.example.com
  authentication:
    "k8s-service-account-tokens":
      kubernetesTokenReview:
        audiences:
        - https://kubernetes.default.svc
  authorization:
    "k8s-rbac":
      kubernetesSubjectAccessReview:
        user:
          selector: auth.identity.user.username
EOF

Replace cluster.example.com with your actual cluster domain.

What if pods are not Running?

If any pod shows CrashLoopBackOff, ImagePullBackOff, or Pending:

  1. Check pod events:

    oc describe pod <pod-name> -n <namespace>

  2. Check pod logs:

    oc logs <pod-name> -n <namespace>

  3. Common issues:

    • Image pull errors: Network connectivity to quay.io

    • Resource constraints: Insufficient CPU or memory on nodes

    • Configuration errors: Invalid AuthConfig or Kuadrant resources

Knowledge check

Test your understanding of tier governance:

  1. Why are MaaSTier policies created before model deployment, rather than after?

    MaaSTier policies must exist before deployment so the MaaS operator can enforce rate limits and quotas from the first inference request. Creating policies after deployment would allow a window of uncontrolled access.

  2. What happens if a Bronze tier user tries to exceed 20 requests/minute?

    Limitador tracks requests per tier and rejects requests exceeding the rate limit with HTTP 429 Too Many Requests. The user must wait until the next minute window to send more requests.

  3. How does Authorino know which tier a user belongs to?

    Authorino validates the JWT token and extracts the ServiceAccount identity. It then checks which OpenShift groups that ServiceAccount’s user belongs to. The MaaSTier resources map groups to tiers (e.g., chocotech-quality-control → Gold tier).

Troubleshooting

If you encounter issues with governance components:

AuthConfig not found

Issue: oc get authconfig -n redhat-ods-applications returns "No resources found"

Solution:

  1. Verify MaaS was enabled in Module 1:

    oc get datasciencecluster default-dsc -o jsonpath='{.spec.components.modelsAsService.managementState}'

    Should return: Managed

  2. Check MaaS operator logs:

    oc logs -n redhat-ods-operator deployment/rhods-operator | grep -i authorino

  3. Recreate AuthConfig manually using the command in Exercise 5

Authorino pods not Running

Issue: Authorino pods show CrashLoopBackOff or Pending

Solution:

  1. Check pod events:

    oc describe pod -l app=authorino -n redhat-ods-applications

  2. Common causes:

    • Insufficient resources: Check node CPU/memory capacity

    • Image pull errors: Verify network connectivity to quay.io

    • Kuadrant operator not installed: Check Kuadrant operator in openshift-operators

Token authentication returns 401 Unauthorized

Issue: curl with Bearer token returns HTTP 401 instead of HTTP 200

Solution:

  1. Verify token is valid and not expired:

    # Regenerate token
TOKEN=$(oc create token marketing-sentiment-app -n chocotech-models --duration=1h)

  2. Verify RBAC is configured:

    oc get role llm-inference-viewer -n chocotech-models
oc get rolebinding marketing-sentiment-app-binding -n chocotech-models

  3. Check Authorino logs for authentication errors:

    oc logs -l app=authorino -n redhat-ods-applications

MaaSTier resources not created

Issue: oc get maastier returns "error: the server doesn’t have a resource type 'maastier'"

Solution:

  1. Verify MaaS CRDs are installed:

    oc get crd | grep maas

    Should show: maastiers.maas.opendatahub.io

  2. If CRD is missing, verify MaaS is enabled:

    oc get datasciencecluster default-dsc -o jsonpath='{.spec.components.modelsAsService.managementState}'

  3. Check OpenShift AI operator logs:

    oc logs -n redhat-ods-operator deployment/rhods-operator | grep -i maastier
