2.1: Creating organizational structure and tier architecture

Before deploying AI models, you need to establish how ChocoTech’s business units will access the MaaS platform. This starts with creating OpenShift groups that represent organizational structure and mapping them to access tiers.

Understanding tier-based governance

Tier-based governance provides fair resource allocation across business units while preventing any single team from monopolizing GPU resources.

The three-tier model

MaaS uses a three-tier model for resource allocation:

  • Gold tier: Production workloads with highest priority and guaranteed resources

    • Intended for mission-critical applications

    • Premium rate limits and quotas

    • Highest request processing priority

  • Silver tier: Development and moderate usage workloads with balanced resources

    • Intended for development and staging environments

    • Moderate rate limits and quotas

    • Medium request processing priority

  • Bronze tier: Exploratory and low-priority workloads with shared resources

    • Intended for experimentation and testing

    • Conservative rate limits and quotas

    • Lower request processing priority

Why tiers matter for MaaS

Tier-based governance provides several critical capabilities:

  • Fair resource allocation: Prevents Bronze tier users from consuming resources needed by Gold tier production workloads

  • Cost control: Rate limits and quotas cap resource consumption per tier

  • Chargeback reporting: Track and allocate AI infrastructure costs by tier and business unit

  • Quality of service: Gold tier gets higher priority during peak demand

Without tiers, the first team to use the platform could monopolize all available GPU resources, blocking other business units.

Creating OpenShift groups

OpenShift groups provide a way to organize users by department or function and assign permissions at the group level rather than individual users.

Why use groups instead of individual users?

Groups provide several advantages over granting permissions to individual users:

  • Scalability: Add new team members by adding them to groups, not by creating individual role bindings

  • Audit trails: Actions are traceable to business units (e.g., "chocotech-quality-control") rather than individual users

  • Team collaboration: All members of a group share the same permissions

  • Simplified management: Change group permissions once instead of updating permissions for each user

Exercise 1: Create business unit groups

Create groups for each of ChocoTech’s business units:

oc adm groups new chocotech-quality-control

Expected output:

group.user.openshift.io/chocotech-quality-control created
oc adm groups new chocotech-rd

Expected output:

group.user.openshift.io/chocotech-rd created
oc adm groups new chocotech-marketing

Expected output:

group.user.openshift.io/chocotech-marketing created

You’ve now created three groups representing ChocoTech’s organizational structure.

Exercise 2: Add yourself to a group

For this lab, you’ll work as a member of the Quality Control team (Gold tier):

oc adm groups add-users chocotech-quality-control $(oc whoami)

Expected output:

group.user.openshift.io/chocotech-quality-control added: "user1"

Verify your group membership:

oc get group chocotech-quality-control -o yaml

Expected output:

apiVersion: user.openshift.io/v1
kind: Group
metadata:
  name: chocotech-quality-control
  resourceVersion: "123456"
  uid: abc123...
users:
- user1

You should see your username in the users list.

Preparing the model serving namespace

Models will be deployed in a dedicated namespace with proper labels for MaaS governance.

Exercise 3: Create and label the namespace

Create a new namespace (project) for model serving:

oc new-project chocotech-models

Expected output:

Now using project "chocotech-models" on server "https://api.cluster.example.com:6443".

You can add applications to this project with the 'new-app' command.

Label the namespace to enable MaaS governance:

oc label namespace chocotech-models maas.opendatahub.io/managed=true

Expected output:

namespace/chocotech-models labeled
oc label namespace chocotech-models modelmesh-enabled=false

Expected output:

namespace/chocotech-models labeled

Understanding namespace labels

These labels serve specific purposes:

  • maas.opendatahub.io/managed=true: Enables MaaS governance for this namespace

    • MaaS tier policies will apply to models in this namespace

    • Rate limits and quotas will be enforced

    • Models can be published to the Gen AI Studio catalog

  • modelmesh-enabled=false: Ensures KServe single-model serving (not ModelMesh)

    • MaaS uses KServe with LLMInferenceService for large language models

    • ModelMesh is designed for serving many small models (not LLMs)

    • This label prevents ModelMesh from managing models in this namespace

Exercise 4: Grant group access to namespace

Grant the Quality Control group admin access to the namespace:

oc create rolebinding chocotech-qc-admin \
  --clusterrole=admin \
  --group=chocotech-quality-control \
  -n chocotech-models

Expected output:

rolebinding.rbac.authorization.k8s.io/chocotech-qc-admin created

This RoleBinding grants members of the chocotech-quality-control group (including you) full admin permissions in the chocotech-models namespace, including:

  • Deploying models (creating LLMInferenceService resources)

  • Managing resources (pods, services, config maps)

  • Viewing logs and metrics

  • Managing RBAC within the namespace

In production, you would create separate RoleBindings for each business unit group with appropriate permissions. For this lab, you’re working as Quality Control (Gold tier) with admin access.

Business value

By mapping ChocoTech’s organizational structure to OpenShift groups and tiers, you’ve established the foundation for:

  • Department-level access control: Quality Control team members can collaborate on model deployment without individual permission management

  • Tier-based resource allocation: Gold tier will receive priority when governance policies are applied in Section 2.3

  • Audit trails: All actions are traceable to business unit groups, not individual users

  • Scalable onboarding: New team members can be added to groups without complex RBAC configuration

When the CIO asks "How do we ensure Quality Control’s production workloads aren’t impacted by Marketing’s experiments?", you can now point to the tier-based group structure.

Knowledge check

Test your understanding before moving to the next section:

  1. Why use OpenShift groups instead of granting access to individual users?

    Groups provide scalability (easy to add new users), audit trails (actions traced to business units), and simplified permission management (change group permissions once instead of per user).

  2. What is the benefit of the maas.opendatahub.io/managed=true label?

    This label enables MaaS governance for the namespace, allowing tier policies, rate limits, and quotas to be enforced on models deployed in this namespace.

  3. Which ChocoTech business unit will have the highest priority for model inference requests?

    Quality Control (Gold tier) will have highest priority (priority: 100) to ensure production defect detection workloads get guaranteed resources.
