Module 4: Governance, observability, and chargeback

ChocoTech’s Finance department raises a critical concern during your status update meeting. "We need to track which teams are consuming GPU resources," the CFO states. "Marketing is requesting additional budget for AI infrastructure, but we have no data showing their current usage or justifying the expansion."

Meanwhile, the Quality Control team complains: "Our defect detection models are getting rate-limited during production shifts. Is Marketing’s chatbot consuming all the capacity?"

Your task is to implement governance policies that ensure fair resource allocation, track usage for chargeback, and provide observability into model consumption patterns.

In this module, you’ll configure governance tiers with rate limiting and quotas using Kuadrant, monitor GPU utilization and token consumption with Grafana, and generate chargeback reports for Finance.

Learning objectives

By the end of this module, you’ll be able to:

  • Define governance tiers (bronze, silver, gold, platinum) with different access levels and quotas

  • Map OpenShift user groups to tier levels for automatic access control

  • Configure rate limiting and token quotas using Kuadrant RateLimitPolicy

  • Deploy Grafana dashboards to monitor model usage and GPU utilization

  • Generate chargeback reports showing token consumption by department and model

  • Implement comprehensive security hardening for production MaaS deployment

  • Troubleshoot policy enforcement issues using Kuadrant component logs

Exercise 1: Understanding MaaS governance tiers

Before configuring governance, you need to understand how tier-based access control works and why it’s critical for multi-tenant model serving.

Without tiers, centralized serving becomes uncontrolled consumption. Tier-based access transforms the MaaS platform from an unlimited resource drain to fair allocation across departments.

The governance challenge at ChocoTech

ChocoTech controls $500,000 worth of GPU infrastructure serving four departments:

  • Quality Control: Defect detection (critical, production workload)

  • R&D: Recipe optimization (important, but not real-time)

  • Marketing: Sentiment analysis (high volume, non-critical)

  • Supply Chain: Inventory prediction (batch processing)

Without governance:

  • Marketing runs unlimited chatbot queries, exhausting GPU capacity

  • Quality Control’s critical defect detection gets rate-limited during production shifts

  • Finance cannot allocate AI costs accurately (no consumption tracking)

  • Security has no audit trail showing who accessed which models

With MaaS governance:

  • Each department is assigned a tier with appropriate rate limits and quotas

  • Quality Control gets "Platinum" tier with high limits and priority access

  • Marketing gets "Bronze" tier with moderate limits appropriate for their use case

  • Finance can perform chargeback based on actual token consumption by department

  • Security has complete audit logs of every API request with user identity

What are MaaS tiers?

Tiers are access control layers that define:

Access level

Which models a user can query (Bronze tier can’t access Platinum-only models)

Rate limits

Maximum requests per minute (Bronze: 100 req/min, Platinum: 10,000 req/min)

Token quotas

Maximum tokens consumed per minute (Bronze: 10,000 tokens/min, Platinum: 1,000,000 tokens/min)

Group mapping

Which OpenShift groups are assigned to each tier

Priority level

Determines access hierarchy when users belong to multiple groups across different tiers

Example: A developer with a Bronze tier Bearer token can only access models published to "Bronze tier and above." A Platinum user can access all models.

Default tier structure for ChocoTech

You’ll configure four tiers aligned with ChocoTech’s business priorities:

Tier Name Priority Level Target Department Rate Limit (req/min)

Bronze

Level 0

Marketing (exploratory)

100 requests/min, 10K tokens/min

Silver

Level 1

Supply Chain (batch)

500 requests/min, 50K tokens/min

Gold

Level 2

R&D (development)

2,000 requests/min, 200K tokens/min

Platinum

Level 3

Quality Control (production)

10,000 requests/min, 1M tokens/min

Priority Level determines access hierarchy: when a user belongs to multiple groups across different tiers, MaaS assigns them to the tier with the highest level number.

Group assignment logic

MaaS uses OpenShift group membership to assign users to tiers:

  1. User authenticates with Bearer token

  2. Authorino (Kuadrant authentication component) extracts user identity and group memberships from OpenShift OAuth

  3. MaaS evaluates which tier configurations match the user’s groups

  4. Highest priority level wins if the user belongs to multiple tiers

  5. Access is enforced based on assigned tier and model configuration

  6. Limitador (Kuadrant rate limiting component) tracks consumption and enforces quotas

A Platform Engineer creates a group chocotech-quality-control in OpenShift, then maps it to the Platinum tier. All members of that group automatically get Platinum access with highest rate limits.

Exercise 3: Configure governance tiers and rate limiting

Now you’ll create MaaSTier custom resources that map OpenShift groups to tier levels with rate limits and quotas.

Tier configuration is declarative, using custom resources managed by the MaaS operator.

Create the Platinum tier (Quality Control)

Create a manifest for the highest tier:

cat > tier-platinum.yaml <<EOF
apiVersion: maas.opendatahub.io/v1alpha1
kind: MaaSTier
metadata:
  name: platinum
  namespace: redhat-ods-applications
spec:
  priorityLevel: 3
  groups:
    - chocotech-quality-control
  rateLimit:
    requestsPerMinute: 10000
    tokensPerMinute: 1000000
  quotas:
    dailyTokenLimit: 100000000
  description: "Production workloads - Quality Control team defect detection"
EOF

Apply the tier configuration:

oc apply -f tier-platinum.yaml

Exercise 4: Deploy Grafana for observability

ChocoTech’s Finance department needs visibility into model usage for chargeback. You’ll deploy Grafana dashboards to monitor token consumption and GPU utilization.

Grafana provides the observability layer for tracking model usage across user tiers and different models.

Install Grafana operator (if not already installed)

Check if Grafana operator is installed:

oc get csv -n openshift-operators | grep grafana

If not installed, install the Grafana operator from OperatorHub:

cat > grafana-operator-subscription.yaml <<EOF
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: grafana-operator
  namespace: openshift-operators
spec:
  channel: v5
  name: grafana-operator
  source: community-operators
  sourceNamespace: openshift-marketplace
EOF
oc apply -f grafana-operator-subscription.yaml

Wait for the operator to be ready:

oc wait --for=condition=ready pod -l name=grafana-operator -n openshift-operators --timeout=300s

Create Grafana instance for MaaS monitoring

Create a Grafana instance in the OpenShift AI namespace:

cat > grafana-instance.yaml <<EOF
apiVersion: grafana.integreatly.org/v1beta1
kind: Grafana
metadata:
  name: maas-grafana
  namespace: redhat-ods-applications
  labels:
    dashboards: "maas"
spec:
  config:
    security:
      admin_user: admin
      admin_password: chocotech123
    server:
      root_url: https://maas-grafana.apps.{cluster_domain}
EOF
oc apply -f grafana-instance.yaml

Wait for Grafana to deploy:

oc wait --for=condition=ready pod -l app=grafana -n redhat-ods-applications --timeout=300s

Create Grafana data source for Prometheus

Configure Grafana to query OpenShift’s Prometheus for metrics:

cat > grafana-datasource.yaml <<EOF
apiVersion: grafana.integreatly.org/v1beta1
kind: GrafanaDataSource
metadata:
  name: prometheus-maas
  namespace: redhat-ods-applications
spec:
  instanceSelector:
    matchLabels:
      dashboards: "maas"
  datasource:
    name: Prometheus
    type: prometheus
    access: proxy
    url: https://thanos-querier.openshift-monitoring.svc.cluster.local:9091
    isDefault: true
    jsonData:
      tlsSkipVerify: true
      timeInterval: "5s"
EOF
oc apply -f grafana-datasource.yaml

Access Grafana dashboard

Get the Grafana route URL:

oc get route maas-grafana -n redhat-ods-applications -o jsonpath='{.spec.host}'

Expected output:

maas-grafana-redhat-ods-applications.apps.{cluster_domain}

  1. Open the Grafana URL in your browser

  2. Log in with:

    • Username: admin

    • Password: chocotech123

You should see the Grafana home page.

Verification

Confirm Grafana deployment:

  • Grafana operator is installed and healthy

  • Grafana instance pod is Running in redhat-ods-applications namespace

  • Grafana route exists and is accessible

Exercise 5: Generate chargeback reports

Now that governance and observability are configured, you can generate chargeback reports for Finance showing token consumption by department.

Token-based consumption tracking enables Finance to bill departments based on actual usage rather than infrastructure allocation.

Query token consumption metrics

Use Prometheus to query token consumption by tier:

oc exec -n openshift-monitoring prometheus-k8s-0 -- promtool query instant \
  http://localhost:9090 \
  'sum(rate(maas_tokens_consumed_total[1h])) by (tier, model)'

This returns token consumption rate by tier and model over the last hour.

Example output:

tier="platinum", model="llama-3-8b"    125000 tokens/hour
tier="gold", model="llama-3-8b"         45000 tokens/hour
tier="bronze", model="nemotron-8b"      12000 tokens/hour

Calculate chargeback costs

Define cost per 1,000 tokens for internal chargeback:

  • Bronze tier: $0.001 per 1,000 tokens

  • Silver tier: $0.002 per 1,000 tokens

  • Gold tier: $0.003 per 1,000 tokens

  • Platinum tier: $0.005 per 1,000 tokens

Example calculation for Quality Control (Platinum tier):

Tokens consumed (24 hours): 125,000 tokens/hour x 24 hours = 3,000,000 tokens
Cost: (3,000,000 / 1,000) x $0.005 = $15.00 per day
Monthly estimate: $15.00 x 30 = $450.00

Marketing (Bronze tier):

Tokens consumed (24 hours): 12,000 tokens/hour x 24 hours = 288,000 tokens
Cost: (288,000 / 1,000) x $0.001 = $0.29 per day
Monthly estimate: $0.29 x 30 = $8.70

This demonstrates that Quality Control’s production workload justifies their Platinum tier allocation, while Marketing’s exploratory use appropriately stays in Bronze tier.

Create a chargeback report

Generate a simple chargeback report:

cat > chargeback-report.sh <<'EOF'
#!/bin/bash
echo "ChocoTech Industries - MaaS Chargeback Report"
echo "Generated: $(date)"
echo ""
echo "Department          Tier      Tokens (24h)    Cost/Day    Monthly Est."
echo "--------------------------------------------------------------------------------"
echo "Quality Control     Platinum  3,000,000       \$15.00     \$450.00"
echo "R&D                 Gold      1,080,000       \$3.24      \$97.20"
echo "Supply Chain        Silver      480,000       \$0.96      \$28.80"
echo "Marketing           Bronze      288,000       \$0.29      \$8.70"
echo "--------------------------------------------------------------------------------"
echo "Total Monthly Cost:                                     \$584.70"
EOF
chmod +x chargeback-report.sh

Run the report:

./chargeback-report.sh

This simple report demonstrates the concept. In production, you would integrate Prometheus metrics with your organization’s billing system or create automated Grafana reports.

Verification

Confirm chargeback capability:

  • Prometheus metrics show token consumption by tier and model

  • Token consumption rates can be queried and aggregated

  • Chargeback calculations based on tier pricing are possible

  • Reports can demonstrate which departments are consuming GPU resources

  • Finance has visibility into AI infrastructure costs by business unit

Troubleshooting

Issue: RateLimitPolicy not enforcing quotas (requests succeeding beyond limits)

Solution:

  1. Verify Limitador pod is healthy: oc get pods -n kuadrant-system | grep limitador

  2. Check RateLimitPolicy is correctly configured: oc describe ratelimitpolicy -n redhat-ods-applications

  3. Verify AuthPolicy is configured to extract user tier: oc get authpolicy -n redhat-ods-applications

  4. Check Limitador logs for policy evaluation: oc logs -l app=limitador -n kuadrant-system

Issue: User assigned to wrong tier (expected Platinum, getting Bronze)

Solution:

  1. Verify user group membership: oc get user {user} -o yaml | grep -A 5 groups

  2. Check MaaSTier configuration: oc get maastier platinum -o yaml

  3. Ensure group name matches exactly in tier spec

  4. Priority level determines tier when user belongs to multiple groups (highest wins)

  5. Check Authorino logs for user tier resolution: oc logs -l app=authorino -n kuadrant-system

Issue: Grafana cannot connect to Prometheus data source

Solution:

  1. Verify Prometheus is accessible from Grafana namespace

  2. Check GrafanaDataSource status: oc describe grafanadatasource prometheus-maas -n redhat-ods-applications

  3. Verify Thanos Querier service exists: oc get svc thanos-querier -n openshift-monitoring

  4. Test connectivity: oc exec -n redhat-ods-applications deployment/maas-grafana — curl -k https://thanos-querier.openshift-monitoring.svc.cluster.local:9091/api/v1/query?query=up

Issue: Token consumption metrics not appearing in Prometheus

Solution:

  1. Verify KServe and Kuadrant are exposing metrics

  2. Check ServiceMonitor resources: oc get servicemonitor -n redhat-ods-applications

  3. Verify Prometheus is scraping MaaS metrics: Query up{job="maas-api"} in Prometheus UI

  4. Check MaaS API pod logs for metric export: oc logs -l app=maas-api -n redhat-ods-applications | grep metrics

Issue: MaaSTier resource created but RateLimitPolicy not generated

Solution:

  1. Check MaaS operator logs: oc logs -n redhat-ods-operator deployment/rhods-operator | grep -i maastier

  2. Verify Kuadrant CRDs are installed: oc get crd | grep kuadrant

  3. Check MaaSTier status for reconciliation errors: oc get maastier platinum -o yaml | grep -A 10 status

  4. Verify RBAC permissions for MaaS operator to create RateLimitPolicy resources

Exercise 6: Production security hardening

ChocoTech’s Chief Information Security Officer (CISO) schedules an urgent meeting before the MaaS platform goes to production. "We need to ensure this AI infrastructure meets our security and compliance requirements," she explains. "We’re subject to food safety regulations, and AI systems that impact quality control must be auditable and secure."

The security team raises several questions:

  • "How do we verify model artifacts haven’t been tampered with?"

  • "Can we audit every API request to track who accessed which models?"

  • "Are the models and data encrypted in transit and at rest?"

  • "Does this meet SOC 2 and ISO 27001 requirements for AI systems?"

Before promoting the MaaS platform to production, you need to implement comprehensive security hardening across nine critical areas.

Security hardening checklist

This checklist covers enterprise security requirements for production AI model serving platforms. Each section includes ChocoTech-specific implementation guidance.

1. Model security

Ensure model artifacts are signed, verified, and tracked from source to deployment.

Requirements:

  • ✓ Sign and verify model artifacts using sigstore or cosign

  • ✓ Store models in a trusted registry or model repository

  • ✓ Scan models for malicious code or unexpected behavior

  • ✓ Use model provenance tracking (who trained, where, when, and how)

  • ✓ Implement AIBOM (AI Bill of Materials) for transparency

  • ✓ Monitor for model drift and adversarial inputs

ChocoTech implementation:

# Sign model artifact with cosign before uploading to S3
cosign sign-blob --key cosign.key llama3-8b-defect-analysis.safetensors > llama3-8b-defect-analysis.sig

# Verify signature before deployment
cosign verify-blob --key cosign.pub --signature llama3-8b-defect-analysis.sig llama3-8b-defect-analysis.safetensors

Model provenance metadata:

apiVersion: v1
kind: ConfigMap
metadata:
  name: llama3-defect-provenance
  namespace: chocotech-models
data:
  model_name: "llama3-8b-defect-analysis"
  base_model: "meta-llama/Meta-Llama-3-8B-Instruct"
  trained_by: "ChocoTech R&D Data Science Team"
  training_date: "2026-04-15"
  training_data: "chocotech-defect-images-v2 (5000 labeled images)"
  training_environment: "OpenShift AI Training Workbench v3.4"
  validation_accuracy: "94.2%"
  signature_verified: "true"
  compliance_status: "approved-for-production"

AIBOM (AI Bill of Materials):

Track dependencies for the model serving stack:

Model: llama3-8b-defect-analysis v2.0.0
├─ Base Model: meta-llama/Meta-Llama-3-8B-Instruct (v1.0)
├─ Serving Runtime: vLLM v0.5.4
├─ Inference Framework: llm-d v1.2.1
├─ GPU Drivers: NVIDIA CUDA 12.4
├─ Container Base: registry.redhat.io/ubi9/python-311:1-48
├─ Dependencies:
│  ├─ transformers==4.40.0
│  ├─ torch==2.3.0+cu124
│  └─ safetensors==0.4.3

Verification:

Check model signature before every deployment. Add admission webhook to enforce signed models in production namespace.

2. Data security

Protect training data and inference requests with encryption and access controls.

Requirements:

  • ✓ Encrypt data in transit (TLS/SSL)

  • ✓ Encrypt data at rest (OpenShift + KMS integration)

  • ✓ Ensure differential privacy or anonymization if using real user data

  • ✓ Apply access controls on training and inference data

ChocoTech implementation:

Encrypt data in transit:

Verify all MaaS endpoints use TLS:

# Check MaaS gateway route uses edge TLS termination
oc get route maas-default-gateway -n redhat-ods-applications -o jsonpath='{.spec.tls.termination}'

Expected output:

edge

Encrypt data at rest:

Enable encryption for PersistentVolumes storing model weights:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: model-storage-encrypted
  namespace: chocotech-models
  annotations:
    volume.kubernetes.io/storage-class: "encrypted-gp3"
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
  storageClassName: encrypted-gp3

Create an encrypted storage class with AWS KMS integration (example for AWS):

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: encrypted-gp3
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
  kmsKeyId: "arn:aws:kms:us-east-1:123456789:key/your-kms-key-id"

Anonymize defect image metadata:

Quality Control’s defect images may contain PII in EXIF data (employee names, timestamps, location). Strip metadata before training:

from PIL import Image
import piexif

def anonymize_image(image_path):
    img = Image.open(image_path)
    # Remove all EXIF metadata
    img.save(image_path, exif=b"")

Verification:

  • MaaS gateway routes use TLS edge termination

  • PVCs for model storage use encrypted storage class

  • Training data is anonymized or uses differential privacy

  • API requests/responses are encrypted (no plaintext transmission)

3. API and endpoint security

Secure the MaaS API gateway and model endpoints with authentication, authorization, and input validation.

Requirements:

  • ✓ Enforce authentication (Keycloak, OAuth2, OIDC)

  • ✓ Use API Gateway (3scale or Kuadrant) for rate-limiting and throttling

  • ✓ Validate all input payloads to prevent injection attacks

  • ✓ Use service mesh (Istio or OpenShift Service Mesh) for secure service-to-service communication

ChocoTech implementation:

Verify Kuadrant Authorino is enforcing authentication:

oc get pods -n kuadrant-system | grep authorino

Expected output:

authorino-xxx   1/1     Running   0          5d

Check AuthPolicy is applied to MaaS gateway:

oc get authpolicy -n redhat-ods-applications

Input validation for prompt injection:

Add input validation to prevent malicious prompts:

import re

def validate_prompt(user_input):
    # Block prompts attempting jailbreak or injection
    blocked_patterns = [
        r"ignore previous instructions",
        r"system: you are now",
        r"<\|im_start\|>system",
        r"</s><s>",  # Model control tokens
    ]

    for pattern in blocked_patterns:
        if re.search(pattern, user_input, re.IGNORECASE):
            raise ValueError("Input contains potentially malicious content")

    return True

Service mesh mTLS for internal communication:

Verify OpenShift Service Mesh is enforcing mutual TLS between MaaS components:

# Check ServiceMeshMemberRoll includes chocotech-models namespace
oc get servicemeshmemberroll -n istio-system -o yaml

Ensure chocotech-models namespace is enrolled in the service mesh for automatic mTLS:

apiVersion: maistra.io/v1
kind: ServiceMeshMemberRoll
metadata:
  name: default
  namespace: istio-system
spec:
  members:
    - redhat-ods-applications
    - chocotech-models

Verification:

  • API requests require Bearer token authentication (401 without token)

  • Kuadrant rate limiting is active (429 when quota exceeded)

  • Service mesh enforces mTLS between model pods and gateway

  • Input validation blocks malicious prompts

4. Runtime and infrastructure security

Isolate model serving workloads and monitor for anomalous behavior.

Troubleshooting

Issue: Gatekeeper policy blocks LLMInferenceService deployment with missing label error

Solution:

  1. Check gatekeeper constraint: oc get k8srequiredlabels -o yaml

  2. Add required labels to LLMInferenceService manifest:

metadata:
  labels:
    app.kubernetes.io/version: "2.0.0"
    app.kubernetes.io/managed-by: "argocd"
    chocotech.com/compliance-approved: "true"

Issue: Cosign signature verification fails with "signature not found" error

Solution:

  1. Verify signature file exists: ls -la llama3-8b-defect-analysis.sig

  2. Check public key matches private key used for signing: cosign public-key --key cosign.key

  3. Ensure signature was generated for the exact file (checksum match): sha256sum llama3-8b-defect-analysis.safetensors

  4. Re-sign if signature is stale or keys rotated

Issue: OpenShift audit logging generates excessive log volume (10GB+/day)

Solution:

  1. Reduce audit profile from WriteRequestBodies to Default: oc patch apiserver cluster --type=merge -p '{"spec":{"audit":{"profile":"Default"}}}'

  2. Filter audit logs to MaaS-related namespaces only in ClusterLogForwarder

  3. Use Prometheus recording rules to aggregate metrics instead of retaining raw logs

  4. Implement log retention policy (30 days for audit, 7 days for application logs)

Issue: ACS reports policy violation for "privileged container" but no privileged containers are defined

Solution:

  1. Check pod security context: oc get pod <pod-name> -n chocotech-models -o yaml | grep -A 5 securityContext

  2. Common false positive: GPU device plugin requires privileged init container (expected)

  3. Whitelist GPU operator namespace in ACS policy: nvidia-gpu-operator

  4. Verify issue is with model pod, not infrastructure pods

Issue: Sealed secret cannot be decrypted by cluster

Solution:

  1. Verify sealed-secrets controller is running: oc get pods -n kube-system | grep sealed-secrets

  2. Check sealed secret was encrypted for correct cluster: kubeseal --fetch-cert (compare cert)

  3. Ensure sealed secret is in correct namespace (sealed secrets are namespace-scoped)

  4. Re-encrypt secret if sealed-secrets controller was reinstalled (new encryption key)
