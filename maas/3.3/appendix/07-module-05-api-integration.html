Module 5: API integration and consumer experience

cat <<EOF | oc apply -f - apiVersion: kuadrant.io/v1beta1 kind: Kuadrant metadata: name: kuadrant namespace: redhat-ods-applications EOF

A developer from ChocoTech’s Marketing team approaches you with excitement. "We’ve been building a customer sentiment analysis application using OpenAI’s ChatGPT API," she explains. "Can we switch to using the internal MaaS models instead? We want to keep customer data on-premises and reduce external API costs."

This is exactly the use case MaaS enables. Your task is to show the development team how to generate API tokens, integrate models using the OpenAI-compatible API, and implement proper error handling for rate limits.

In this module, you’ll navigate the Gen AI Studio interface, generate Bearer tokens for authentication, consume models via the OpenAI-compatible API, and implement retry logic for production applications.

Learning objectives

By the end of this module, you’ll be able to:

  • Navigate the Gen AI Studio dashboard to discover available models

  • Generate API tokens with proper scope and expiration settings

  • Make REST API calls to /v1/models and /v1/chat/completions endpoints

  • Implement OpenAI-compatible API integration in existing applications

  • Handle rate limiting responses (429 status) with exponential backoff retry logic

  • Understand token consumption tracking and quota management

Exercise 2: Consume models via OpenAI-compatible API

Now that you have a Bearer token, you’ll make REST API calls to list models and generate chat completions.

MaaS exposes models through the OpenAI-compatible API, enabling existing applications built for ChatGPT or GPT-4 to switch to self-hosted models by changing the base URL and Bearer token.

Set up environment variables

Save your Bearer token as an environment variable:

export MAAS_TOKEN="<paste-your-token-here>"

Replace <paste-your-token-here> with the actual token you copied from Gen AI Studio.

Get the MaaS gateway URL:

export MAAS_URL=$(oc get route maas-default-gateway -n redhat-ods-applications -o jsonpath='{.spec.host}')

Verify the environment variables:

echo "MaaS Gateway: https://${MAAS_URL}"
echo "Token starts with: ${MAAS_TOKEN:0:20}..."

Expected output:

MaaS Gateway: https://maas-default-gateway.apps.{cluster_domain}
Token starts with: eyJhbGciOiJSUzI1Ni...

List available models

Query the /v1/models endpoint to discover available models:

curl -s -X GET "https://${MAAS_URL}/v1/models" \
  -H "Authorization: Bearer ${MAAS_TOKEN}" \

Exercise 3: Implement retry logic for rate limiting

Production applications must handle rate limiting responses (429 status) gracefully using exponential backoff retry logic.

When users exceed their tier’s rate limit or quota, Kuadrant returns 429 Rate Limit Exceeded with a Retry-After header indicating when to retry.

Understand rate limit responses

Make multiple rapid requests to trigger rate limiting (Bronze tier: 100 req/min):

for i in {1..10}; do
  curl -s -w "\nHTTP Status: %{http_code}\n" -X POST "https://${MAAS_URL}/v1/chat/completions" \
    -H "Authorization: Bearer ${MAAS_TOKEN}" \
    -H "Content-Type: application/json" \
    -d '{
      "model": "meta-llama/Meta-Llama-3-8B-Instruct",
      "messages": [{"role": "user", "content": "Hi"}],
      "max_tokens": 10
    }' | head -5
  sleep 0.5
done

If you exceed rate limits, you’ll see:

HTTP Status: 429

{
  "error": {
    "message": "Rate limit exceeded. Retry after 45 seconds.",
    "type": "rate_limit_exceeded",
    "code": 429
  }
}

The response includes:

  • HTTP status 429 Rate Limit Exceeded

  • Retry-After header with seconds to wait

  • Error message explaining the limit

Implement exponential backoff retry logic

Create a Python script with proper retry logic:

cat > maas_client_with_retry.py <<'EOF'
#!/usr/bin/env python3
import os
import time
import requests
from typing import Optional

MAAS_URL = f"https://{os.getenv('MAAS_URL')}"
MAAS_TOKEN = os.getenv('MAAS_TOKEN')

def chat_completion_with_retry(
    model: str,
    messages: list,
    max_retries: int = 3,
    initial_backoff: float = 1.0
) -> Optional[dict]:
    """
    Make chat completion request with exponential backoff retry logic.

    Args:
        model: Model identifier
        messages: List of chat messages
        max_retries: Maximum number of retry attempts
        initial_backoff: Initial backoff delay in seconds

    Returns:
        API response dict or None if all retries failed
    """
    headers = {
        "Authorization": f"Bearer {MAAS_TOKEN}",
        "Content-Type": "application/json"
    }

    payload = {
        "model": model,
        "messages": messages,
        "max_tokens": 200
    }

    backoff = initial_backoff

    for attempt in range(max_retries + 1):
        try:
            response = requests.post(
                f"{MAAS_URL}/v1/chat/completions",
                headers=headers,
                json=payload,
                timeout=30
            )

            # Success
            if response.status_code == 200:
                return response.json()

            # Rate limit exceeded
            elif response.status_code == 429:
                if attempt < max_retries:
                    # Get retry-after header if present, otherwise use exponential backoff
                    retry_after = int(response.headers.get('Retry-After', backoff))
                    print(f"Rate limit exceeded. Retrying after {retry_after} seconds (attempt {attempt + 1}/{max_retries})")
                    time.sleep(retry_after)
                    backoff *= 2  # Exponential backoff for next attempt
                else:
                    print(f"Rate limit exceeded. Max retries ({max_retries}) reached.")
                    return None

            # Unauthorized or forbidden
            elif response.status_code in [401, 403]:
                print(f"Authentication error: {response.status_code}")
                print(response.json())
                return None

            # Other errors
            else:
                print(f"API error: {response.status_code}")
                print(response.json())
                return None

        except requests.exceptions.Timeout:
            print(f"Request timeout (attempt {attempt + 1}/{max_retries})")
            if attempt < max_retries:
                time.sleep(backoff)
                backoff *= 2
        except requests.exceptions.RequestException as e:
            print(f"Request failed: {e}")
            return None

    return None

# Test the retry logic
if __name__ == "__main__":
    print("Testing MaaS API with retry logic...\n")

    result = chat_completion_with_retry(
        model="meta-llama/Meta-Llama-3-8B-Instruct",
        messages=[
            {"role": "system", "content": "You are a helpful assistant for ChocoTech Industries."},
            {"role": "user", "content": "What are the benefits of dark chocolate?"}
        ],
        max_retries=3,
        initial_backoff=1.0
    )

    if result:
        print("Success!")
        print(f"Response: {result['choices'][0]['message']['content']}")
        print(f"\nTokens used: {result['usage']['total_tokens']}")
    else:
        print("Failed to get response after retries.")
EOF

Exercise 4: OpenAI API compatibility and migration patterns

MaaS uses the OpenAI-compatible API format, making it easy to migrate existing applications from OpenAI to self-hosted models.

Applications built for ChatGPT or GPT-4 can switch to MaaS models by changing two parameters: base URL and API key.

OpenAI SDK compatibility

Install the OpenAI Python SDK:

pip install openai

Create a migration example showing how to switch from OpenAI to MaaS:

cat > openai_to_maas_migration.py <<'EOF'
#!/usr/bin/env python3
import os
from openai import OpenAI

# BEFORE: Using OpenAI ChatGPT
# client = OpenAI(
#     api_key=os.getenv("OPENAI_API_KEY")
# )

# AFTER: Using MaaS (only two changes required)
client = OpenAI(
    base_url=f"https://{os.getenv('MAAS_URL')}/v1",  # Change 1: MaaS gateway URL
    api_key=os.getenv("MAAS_TOKEN")                   # Change 2: MaaS Bearer token
)

# The rest of the code remains identical
response = client.chat.completions.create(
    model="meta-llama/Meta-Llama-3-8B-Instruct",  # Use MaaS model instead of gpt-4
    messages=[
        {"role": "system", "content": "You are a helpful assistant for ChocoTech Industries."},
        {"role": "user", "content": "How can we reduce production waste in chocolate manufacturing?"}
    ],
    max_tokens=200,
    temperature=0.7
)

print("Response from MaaS model:")
print(response.choices[0].message.content)
print(f"\nTokens used: {response.usage.total_tokens}")
EOF

  1. Replace https://api.openai.com/v1 with https://<maas-gateway>/v1

Troubleshooting

Issue: API returns 401 Unauthorized

Solution:

  1. Verify Bearer token is valid and not expired in Gen AI Studio → API tokens

  2. Check token is correctly set in environment variable: echo ${MAAS_TOKEN:0:20}

  3. Ensure Authorization header format: Authorization: Bearer <token> (not Bearer: <token>)

  4. Verify token was copied correctly (no extra spaces or newlines)

Issue: API returns 403 Forbidden

Solution:

  1. Your tier does not have access to the requested model

  2. Check model’s tier access in Gen AI Studio → Models tab

  3. Verify your user group membership: oc get user {user} -o yaml | grep groups

  4. Check which tier your group is assigned to

  5. Request access from Platform Engineer to upgrade tier or publish model to your tier

Issue: API returns 429 Rate Limit Exceeded immediately

Solution:

  1. You’ve exhausted your tier’s quota (requests/min or tokens/min)

  2. Check usage in Gen AI Studio → Usage tab

  3. Implement exponential backoff retry logic (see Exercise 3)

  4. Request tier upgrade from Platform Engineer if usage is legitimate

  5. Check Retry-After header for when quota resets

Issue: OpenAI SDK raises "Connection refused" error

Solution:

  1. Verify MaaS gateway route exists: oc get route maas-default-gateway -n redhat-ods-applications

  2. Check MAAS_URL environment variable is correct: echo $MAAS_URL

  3. Verify network connectivity from client to OpenShift cluster

  4. Check for firewall or proxy blocking HTTPS traffic

  5. Test with curl first to isolate SDK vs connectivity issues

Issue: Model inference returns empty or truncated response

Solution:

  1. Check max_tokens parameter - may be too low for complete response

  2. Increase max_tokens to allow longer completions (e.g., 500 or 1000)

  3. Check finish_reason in response - if "length", tokens limit was hit

  4. Verify model is healthy: oc get llminferenceservice -n chocotech-models

  5. Check model pod logs for inference errors: oc logs -l app=llama-3-8b -n chocotech-models

Issue: Requests timing out after 30 seconds

Solution:

  1. Large models or long prompts may take time to generate responses

  2. Increase request timeout in client code (e.g., timeout=60)

  3. Check model pod CPU/GPU utilization - may be overloaded

  4. Consider using streaming responses for better user experience

  5. Check for model pod issues: oc get pods -n chocotech-models
