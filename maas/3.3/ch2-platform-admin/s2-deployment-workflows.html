Deployment Workflows

Estimated reading time: 10 minutes.

Two paths to MaaS: build the platform first then add models, or enroll existing llm-d models into MaaS.

Objective

Understand the platform-first vs model-retrofit deployment workflows and when to use each approach.

Platform-First Deployment

The platform-first approach enables MaaS infrastructure before deploying any models:

1. Enable MaaS (DataScienceCluster CRD)
2. Verify platform components are healthy
3. Deploy new models with "Publish as MaaS endpoint" enabled
4. Models automatically appear in catalog with governance

This is the recommended approach for new deployments.

When to Use:

  • Starting a new OpenShift AI implementation

  • No existing KServe models in production

  • Want to establish governance before enabling inference

Advantages:

  • Clean implementation—all models governed from day one

  • Platform team sets policies before developers consume

  • No migration complexity

Model-Retrofit Deployment

The retrofit approach enrolls existing generative AI models into MaaS.

1. Existing model using the "Distributed inference with llm-d" runtime is already deployed
2. Enable MaaS platform components
3. Edit the existing model deployment through the dashboard
4. Enable "Publish as MaaS endpoint" and assign tier access

When to Use:

  • OpenShift AI cluster already running supported models

  • Need to add governance to existing inference workloads

  • Migrating from standalone serving to centralized MaaS

Currently, only the Distributed inference with llm-d runtime supports MaaS integration. If an existing model uses a different runtime, the option to publish it as a MaaS endpoint will not be available.

Enrollment Process via UI:

Update existing models through the OpenShift AI dashboard:

  1. Centralized Model HubNavigate to Projects and click your project name.

  2. Click the Deployments tab..

  3. In the Advanced settings section, select Publish as MaaS endpoint.

  4. Configure Tier access (select either All tiers or Specific tiers).

  5. Centralized Model Hub: Click Update.

After updating, the model appears in the Gen AI Studio dashboard and becomes accessible through the MaaS gateway.

Enrollment Process via CLI:

Add this annotation to existing LLMInferenceService resources:

apiVersion: serving.kserve.io/v1beta1
kind: LLMInferenceService
metadata:
  name: existing-model
  annotations:
    alpha.maas.opendatahub.io/tiers: '["free", "premium", "enterprise"]'  (1)
spec:
  # ... existing model configuration
1 This annotation enrolls the model in MaaS catalog

After annotation, the model appears in the Gen AI Studio dashboard and becomes accessible through the MaaS gateway.

Migration Considerations

When retrofitting existing models:

Immediate Effect

Changing tier access and MaaS settings takes effect immediately. Users who lose access will receive 403 Forbidden errors on their next API request.

Client Changes

Applications consuming the old KServe endpoint must migrate to the MaaS gateway URL and use Bearer token authentication.

Tier Assignment

Retrofitted models default to "All tiers" access if not explicitly scoped during the update. You may want to restrict access to specific tiers during the enrollment process.

Rate Limits

Models are immediately subject to MaaS rate limits upon enrollment. Monitor for 429 Rate Limit Exceeded errors from existing clients.

Best Practices

For Platform-First:

  • Enable MaaS during initial OpenShift AI setup

  • Document deployment workflows before onboarding teams

  • Create default tiers aligned with organizational structure

For Model-Retrofit:

The retrofit workflow cannot be applied to just any existing KServe model; the model must already be deployed using the "Distributed inference with llm-d" runtime.

  • Test enrollment on non-production models first

  • Communicate gateway URL changes to application teams

  • Migrate models gradually (not all at once)

  • Monitor for client 401/403/429 errors after migration
